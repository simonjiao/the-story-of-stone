# 16 现有架构差距与实施方向

## 目标

本文明确当前仓库距离“通灵玉”第一版目标的差距，并给出实施方向。

本轮判断只面向通灵玉 Agent 本身，不关心 `agent-platform/` 的通用平台实现。通灵玉应作为新的独立 Agent 产品线实现，围绕 Open WebUI 单入口、Gateway 编排、四个内部 Agent、知识库检索、证据包和 reviewer 审校闭环推进。

## 当前仓库已有内容

| 模块 | 当前状态 | 结论 |
|---|---|---|
| 产品设计 | 已导入 `docs/tonglingyu-agent-design/00` 到 `15` | 可作为第一版目标依据 |
| 视频语料 | 已有 `resources/styles/buhongjushi/` 转录和元数据 | 可作为风格资料，不是主证据库 |
| 旧基础产物 | `resources/base/hongloumeng/` 来自旧专用抽取 | 已删除，不作为新知识库输入 |
| EPUB 抽取脚本 | 原脚本绑定红楼梦目录、输出和命名 | 已改造为通用 source snapshot 抽取入口 |
| 知识库服务 | 只有草案文档和预留包 | 尚未实现 |
| 通灵玉 Agent | 尚无 Gateway、profile、工具权限和审校链路实现 | 需要新建 |
| Open WebUI 集成 | 现有部署经验可复用，但没有通灵玉模型入口配置 | 需要单独配置 |

## 与第一版目标的关键差距

| 目标能力 | 设计要求 | 当前差距 | 第一版处理 |
|---|---|---|---|
| 单一可见模型 | Open WebUI 只展示“通灵玉” | 仓库没有通灵玉模型注册和路由 | 新增 Gateway 模型发现与聊天入口 |
| Gateway 编排 | Gateway 不是 Agent，负责鉴权、会话、编排、安全和审计 | 没有通灵玉 Gateway 服务 | 新建独立 Gateway，不复用 `agent-platform` 作为主线 |
| 四个内部 Agent | `honglou-main/text/commentary/reviewer` 分工明确 | 没有 profile 配置、提示词、schema 和工具白名单 | 新增通灵玉 Agent profile 目录和调用契约 |
| 知识库输入 | 使用新资料构造原文、脂批、人物事件、版本说明库 | 旧产物已删除，建库逻辑尚未消费 source snapshot | 先建立来源登记和建库输入格式 |
| 证据卡片 | RAG 只返回可追溯证据卡片 | 没有 evidence card schema 和生成逻辑 | 定义 schema 后由知识服务返回 |
| 证据包 | Gateway 形成证据包并映射结论声明 | 没有 evidence package store | 新增请求级证据包记录 |
| 审校硬流程 | 复杂回答必须经过 reviewer | 没有 reviewer 调用和拦截规则 | Gateway 固定执行 reviewer，不允许用户关闭 |
| 审计治理 | 记录追踪编号、调用顺序、证据包、审校结果 | 没有通灵玉审计表或日志格式 | 新增最小 JSONL/SQLite 审计记录 |
| 验证 | 标准问题集、负面用例、上线门槛 | 尚无通灵玉评测集 | 先建 100 到 200 条第一版问题集 |

## 明确不走的方向

第一版不把 `agent-platform/` 作为通灵玉实现前提。

原因：

1. 通灵玉第一版的核心风险在证据可靠性、知识库输入、Gateway 固定流程和 reviewer 审校，而不是通用 Agent 平台能力。
2. `agent-platform/` 面向通用运行、队列、审批和工作流管理；通灵玉需要更窄、更确定的产品链路。
3. 第一版应先验证“资料入库 -> 证据卡片 -> 证据包 -> 审校 -> 分层回答”的闭环，避免被平台化边界拖慢。

## 新实现边界

建议新增独立实现边界：

| 边界 | 建议职责 |
|---|---|
| `scripts/extract_epub.py` | 通用 EPUB source snapshot 抽取，不再绑定《红楼梦》 |
| `scripts/download_wikisource.py` | 维基文库/MediaWiki source snapshot 下载，用于《红楼梦》全本、脂批本等资料 |
| `src/tonglingyu_agent/` | 通灵玉 Gateway、Agent 调用、证据 schema、审校流程的 Python 实现入口 |
| `resources/sources/` | 新资料的规范化快照和来源元数据 |
| `data/tonglingyu/` | 本地生成的 SQLite、索引、审计和评测运行产物，默认不提交 |
| `docs/tonglingyu-agent-design/` | 产品和架构冻结文档 |

## 第一阶段实施顺序

1. 通用资料抽取：把 EPUB、维基文库全本、脂批资料、版本说明资料抽为可登记 source snapshot。
2. 来源登记：每个资料源必须有 `source_id`、来源类别、版本、许可/出处说明、抽取时间和输入摘要。
3. 知识库建库：从 source snapshot 构造原文库、脂批库、人物事件库和版本说明库。
4. 证据卡片：统一正文、脂批、版本、人物事件证据的返回 schema。
5. 通灵玉 Gateway：实现 OpenAI-compatible 模型发现和聊天入口，只暴露“通灵玉”。
6. Agent profiles：实现 `honglou-main`、`honglou-text`、`honglou-commentary`、`honglou-reviewer` 的提示词、工具白名单和输出 schema。
7. 审校和审计：复杂回答固定经过 reviewer，并写入请求追踪、证据包和审校结果。
8. 验证集：用标准问题集和负面清单验证上线条件。
