# 16 现有架构差距与实施方向

本文只面向通灵玉 Agent 第一版，不把通用平台化能力作为前置条件。第一版目标是验证证据型 RAG 闭环：

`资料入库 -> 证据卡片 -> 证据包 -> reviewer 审校 -> 分层回答`

## 当前状态

| 模块 | 状态 | 处理 |
|---|---|---|
| 产品设计 | `00` 到 `15` 已导入 | 作为第一版目标依据 |
| 风格语料 | `resources/styles/buhongjushi/` 已存在 | 只用于表达风格和待校订转录 |
| 资料抽取 | EPUB 和维基文库脚本已输出 source snapshot | 可开始处理新资料 |
| 字形读音 | 脚本已写入 `rare_char_annotations` | 建库必须消费 |
| 基础资料 | `resources/sources/` 尚无正式快照 | 先下载并登记 |
| 知识库 | SQLite/FTS、八类知识库、证据卡片未实现 | 新建 |
| Agent | Gateway、profiles、reviewer 未实现 | 新建 |
| Open WebUI | 没有“通灵玉”模型入口 | 新增单入口配置 |

旧基础库产物和旧专用抽取脚本已删除，不再作为实现起点。

## 必补能力

| 能力 | 第一版要求 |
|---|---|
| 单一可见模型 | Open WebUI 只展示“通灵玉” |
| Gateway | 负责鉴权、会话、编排、安全、审计和 OpenAI-compatible 入口 |
| 内部 Agent | `honglou-main`、`honglou-text`、`honglou-commentary`、`honglou-reviewer` |
| 知识库 | 从 source snapshot 构造原文、脂批、版本、人物、关系、事件、诗词判词、评测题库和字形读音索引 |
| 证据卡片 | RAG 只返回可追溯证据，不直接生成最终回答 |
| 证据包 | Gateway 汇总证据并映射到结论声明 |
| reviewer | 复杂回答固定审校，不提供用户关闭开关 |
| 审计 | 记录追踪编号、调用顺序、证据包和审校结果 |
| 验证 | 建立 100 到 200 条标准问题和负面用例 |

## Source Snapshot 合同

每个入库来源至少包含 `source_id`、`source_category`、`format`、`title`、`work`、`edition`、`language`、来源 URL 或文件 hash、抽取时间和 `documents/blocks.jsonl`。

允许的 `source_category`：

- `base_material`
- `extended_base_material`
- `commentary_material`
- `research_material`
- `style_material`
- `evaluation_material`

原始字形必须保留；规范化文本只能作为检索辅助字段。来源中已有 ruby、反切、拼音、注音或人工校订读音必须进入 `rare_char_annotations`，并能回溯到具体 block 和来源位置。

## 第一版数据边界

本地生成物写入 `data/tonglingyu/`，默认不提交。最小 SQLite/FTS 结构应覆盖：

- `sources`
- `editions`
- `chapters`
- `blocks`
- `blocks_fts`
- `rare_char_annotations`
- `commentaries`
- `versions`
- `people`
- `aliases`
- `relationships`
- `events`
- `poems`
- `terms`
- `evidence_cards`
- `evaluation_cases`
- `kb_version`

模型生成内容不得反写为知识库事实。风格资料不覆盖正文、脂批、版本或校订证据。现代白话摘要只能辅助检索，不能作为可引用证据。

第一版知识库不是一个大向量库。正文、脂批、版本、人物关系、事件、诗词判词和研究观点必须分层；每张证据卡片必须包含支持范围和不支持范围。

## 实施顺序

1. 下载并登记维基文库《红楼梦》全本、脂批本和版本说明资料。
2. 实现 source snapshot loader。
3. 实现原文、脂批、版本、人物、关系、事件、诗词判词、评测题库和 `rare_char_annotations` 的最小 schema。
4. 实现 FTS、别名索引、脂批/版本索引和反证/限制索引。
5. 定义证据卡片 schema，强制支持范围、不支持范围和校验状态。
6. 建立覆盖正文、脂批、版本、人物别名、诗词判词、字形读音和证据不足的评测集。
7. 实现 Gateway、内部 Agent profiles、证据包、reviewer 和审计。
8. 配置 Open WebUI 只暴露“通灵玉”。
