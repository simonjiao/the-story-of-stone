# 19 第一批资料来源登记

## 文档定位

本文登记“通灵玉”第一版第一批 source snapshot 的来源、版本边界、
使用方式和未决问题。它是 M1 阶段的执行记录，不替代后续知识库
schema、loader 或评测设计。

## 已确认执行决策

1. 第一批资料从维基文库获取；
2. 允许联网下载公开资料；
3. 资料必须能确定版本或来源边界；
4. 第一版不接入外部向量库，只使用 SQLite/FTS；
5. 远程部署复用一个现有 Open WebUI；
6. Gateway 单独部署，不混入 Open WebUI 容器；
7. Open WebUI 作为唯一用户入口，Gateway 作为 OpenAI-compatible 后端。

## M1 完成闸门

本计划不设置独立的“M1.5”。进入 M2 之前只有一个 M1 完成闸门；凡是
要求在 M2 loader 实现前完成的事项，都必须明确列入这个闸门并验收通过。

当前 M1 闸门范围是“第一批 Wikisource source snapshot 工程可入库”，
不包含学术校勘完成声明。通过条件如下：

1. 5 个第一批来源全部存在；
2. 每个来源都有 `metadata/source.json`、抽取报告、页面清单、
   `documents/documents.jsonl` 和 `documents/blocks.jsonl`；
3. 每个来源 `missing` 为 `0`，documents 和 blocks 计数与报告一致；
4. 版本依据以原始 `.wiki`、`revision_id`、`source_url` 和 metadata 为准；
5. 不保存空的 HTML 派生产物；
6. 19 个跨版本抽样点可定位；
7. `scripts/validate_source_snapshots.py` 通过。

独立影印件、权威校注本、研究观点来源、评测题库和证据卡片不在当前
M1 闸门内。它们必须在后续证据校验、评测或发布 QA 节点中单独成闸门；
如果决定把其中任何一项提升为 M1 P0，则 M2 暂停，直到该项完成并通过。

## 命名规则

source snapshot 的 `source_id` 采用稳定 ASCII 命名：

`作品或系统-wikisource-版本或范围`

示例：

1. `hongloumeng-wikisource-120`
2. `hongloumeng-wikisource-chengyi`
3. `shitouji-wikisource-zhiyanzhai`
4. `shitouji-wikisource-jiaxu`

版本说明放入 `metadata/source.json` 的 `edition` 和 `notes`，不再用
拼音后缀表达复杂版本语义。

## Source Snapshot Ready 口径

当前只能声明“第一批 Wikisource snapshot 工程上可进入 loader”，不能声明
“文本已经完成学术校勘”。ready 的最小口径如下：

1. snapshot 目录、metadata、documents 和 blocks 文件存在；
2. `metadata/source.json` 记录 `source_id`、分类、版本边界、来源 URL、
   许可证说明和下载时间；
3. `metadata/extraction_report.json` 中 `missing` 为 `0`；
4. `documents/documents.jsonl` 和 `documents/blocks.jsonl` 行数与抽取报告一致；
5. 每个 block 有稳定 `block_id`、`source_title`、`source_url`、
   `revision_id` 和原始文本；
6. 跨版本抽样点能通过 block ID 和关键字定位；
7. 原始 source snapshot 的空白、异体字、生僻字不为通过 lint 而改写。

明确不声明：

1. 不声明 Wikisource 转录无误；
2. 不声明完成版本校勘；
3. 不声明本批资料是维基文库页面的 HTML 镜像；版本追踪以 `.wiki`、
   `revision_id`、`source_url` 和 metadata 为主；
4. 不声明后续证据卡片或回答已经正确；
5. 不替代后续影印件或权威校注本复核。

机器可验证口径位于：

1. `resources/sources/wiki/_registry/first_batch_acceptance.json`
2. `resources/sources/wiki/_registry/first_batch_samples.jsonl`
3. `scripts/validate_source_snapshots.py`

当前校验命令：

```bash
python3 scripts/validate_source_snapshots.py
```

该命令检查 5 个来源和 19 个抽样点，覆盖目录、第一回石头来源、
第八回通灵玉铭文、八十回边界、后四十回收束、程甲/程乙对照、
甲戌本和脂批整理资料。

`raw_html` 不作为本批 source snapshot 合同的一部分。原因是 HTML 是
MediaWiki 对原始 wikitext 的派生渲染，批量 API 在部分页面上会返回空的
解析结果；保存空 HTML 会制造“已镜像页面”的错觉。当前源码级版本依据是
原始 `.wiki`、页面 `revision_id`、`source_url` 和抽取报告。若后续需要
HTML 渲染层，可使用下载脚本的 `--save-raw-html` 显式生成，并单独验收。

## 来源边界

### `hongloumeng-wikisource-120`

维基文库页面说明该文本为 120 回数字化文本，不包括批语，前八十回以
庚辰本为底本，后四十回以程甲本为底本。

使用边界：

1. 可作为第一版正文主库候选；
2. 回答中必须保留前八十回和后四十回边界；
3. 不能把该来源当作影印校勘依据；
4. 关键引文后续仍可用影印或权威校注本复核。

### `hongloumeng-wikisource-chengyi`

维基文库程乙本数字化文本，页面按十回分段。

使用边界：

1. 可作为程乙本版本对照候选；
2. 不直接覆盖 120 回汇校本正文主库；
3. 涉及版本差异时必须标注“程乙本”。

### `hongloumeng-wikisource-chengjia`

维基文库程甲本数字化文本。该来源子页使用 ProofreadPage 的
`<pages index=...>` transclusion，本批下载已通过 Page namespace 展开
正文，并在 documents 中记录 `proofread_pages`。

使用边界：

1. 可作为程甲本版本对照候选；
2. 涉及版本差异时必须标注“程甲本”；
3. 关键引文仍建议后续用影印或权威校注本复核。

### `shitouji-wikisource-zhiyanzhai`

维基文库整理的《脂硯齋重評石頭記》。

使用边界：

1. 只能作为脂批、版本和评语证据候选；
2. 不得等同正文事实；
3. 必须保留批语来源、版本说明和支持范围；
4. 单条批语不能直接推出确定人物结局。

### `shitouji-wikisource-jiaxu`

维基文库整理的甲戌本脂评资料。

使用边界：

1. 只能作为甲戌本相关脂批和版本证据候选；
2. 覆盖范围有限，回答必须说明版本和回目限制；
3. 不能把甲戌本批语扩展为所有版本的正文事实。

## 程甲本补齐说明

程甲本没有继续使用逐页 `parse` 硬抓。下载脚本已增加 ProofreadPage
展开逻辑：解析 `<pages index=... from=... to=...>`，批量抓取
`Page:<index>/<page>` 原始文本，去除 `noinclude`，按
`fromsection/tosection` 裁剪后生成 blocks。

## 更好的后续来源建议

维基文库适合作为第一批可追溯公开数字文本，但不是最终校勘依据。
后续更好的增强来源包括：

1. 程甲本、程乙本影印件或权威校注本；
2. 庚辰本、甲戌本等脂本影印或校注资料；
3. 可标注页码、卷册、版本系统的学术整理本；
4. 专门的版本说明和红学工具书条目。

这些来源进入知识库前仍必须转为 source snapshot，并保留 hash、来源说明、
抽取时间和使用限制。
