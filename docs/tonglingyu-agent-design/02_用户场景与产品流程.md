# 02 用户场景与产品流程

## 场景分类

第一版将用户问题分为六类：

| 场景 | 示例问题 | 必需证据 |
|---|---|---|
| 原文定位 | 林黛玉第几回进贾府？ | 正文 |
| 诗词判词解释 | 晴雯判词怎么理解？ | 正文、必要时人物事件 |
| 人物关系 | 林黛玉和史湘云以前认识吗？ | 正文、人物关系证据 |
| 人物命运 | 香菱最后死了吗？ | 正文、脂批、版本差异、推断边界 |
| 脂批问题 | 脂批有没有暗示宝玉出家？ | 脂批、对应正文 |
| 版本差异 | 前八十回和后四十回如何看某人物结局？ | 版本信息、正文分层 |

## 用户可见流程

用户看到的流程应保持简单：

1. 用户登录 Open WebUI；
2. 选择“通灵玉”；
3. 提出问题；
4. 系统返回分层回答；
5. 用户可以继续追问证据、版本或人物关系。

用户不应看到：

1. 四个内部 Agent；
2. Gateway 内部编排过程；
3. 工具调用日志；
4. 内部提示词；
5. 系统配置或接口细节。

## 内部处理流程

内部流程是：

1. Gateway 接收 Open WebUI 请求；
2. Gateway 鉴权并归一化请求；
3. `honglou-main` 判断问题类型并生成检索计划；
4. `honglou-text` 检索正文证据；
5. `honglou-commentary` 在需要时检索脂批和版本证据；
6. Gateway 汇总证据包；
7. `honglou-main` 生成草稿；
8. `honglou-reviewer` 审校；
9. 如审校未通过，`honglou-main` 修订；
10. Gateway 返回最终回答。

## 回答展示原则

复杂问题建议使用以下结构：

1. 结论；
2. 正文依据；
3. 脂批依据；
4. 版本差异；
5. 合理推断；
6. 需要谨慎的地方。

简单问题可以简化，但仍应保留可追溯依据。

## 典型流程一：原文定位

用户问：

> 林黛玉第几回进贾府？

内部预期：

1. Gateway 接收请求；
2. `honglou-main` 识别为原文定位问题；
3. `honglou-text` 检索正文；
4. `honglou-main` 组织回答；
5. `honglou-reviewer` 检查回目和证据；
6. 返回答案。

回答应包含：

1. 回目；
2. 简短原文依据或片段说明；
3. 不展开不必要的脂批讨论。

## 典型流程二：人物命运

用户问：

> 香菱最后死了吗？

内部预期：

1. `honglou-main` 识别为人物命运问题；
2. 强制检索正文；
3. 强制检索脂批和版本；
4. 生成证据包；
5. 审校是否把推断说成定论；
6. 返回分层回答。

回答必须区分：

1. 前八十回正文是否直接交代；
2. 后四十回通行本如何处理；
3. 脂批是否有提示；
4. 哪些是研究推断；
5. 哪些结论证据不足。

## 场景验收方式

每个场景至少准备以下材料：

| 材料 | 用途 |
|---|---|
| 标准问题集 | 验证分类是否正确 |
| 必须命中的证据 | 验证检索是否正确 |
| 禁止出现的结论 | 验证负面清单是否生效 |
| 推荐回答形态 | 验证最终展示是否符合产品预期 |

## RAG 场景分流

不同问题类型应触发不同证据来源，而不是统一做语义检索。

| 用户问题类型 | 必需证据来源 | 回答边界 |
|---|---|---|
| 原文定位 | 原文库、回目结构 | 给出位置和原文依据 |
| 诗词判词 | 诗词判词曲文库、原文库 | 区分原文和解释 |
| 人物关系 | 人物库、关系库、正文证据 | 标注明确关系、文本暗示或推断 |
| 人物命运 | 原文库、脂批库、版本库、事件库 | 区分前八十回、后四十回、脂批和推断 |
| 脂批问题 | 脂批库、对应正文、版本库 | 不把脂批当正文事实 |
| 版本差异 | 版本库、对齐正文、来源说明 | 不给无版本标签的结论 |

用户最终看到的是分层回答；内部必须已经完成证据卡片和证据包构造。
