# 14 迭代路线图

## 第一版目标

第一版只做最小可靠闭环：

> Open WebUI 单入口 → Gateway 编排 → 四个内部 Agent → 知识库检索 → 证据包 → 审校 → 分层回答

## 第一版范围

| 模块 | 范围 |
|---|---|
| Open WebUI | 一个可见模型 |
| Gateway | 鉴权、会话映射、固定编排、审计 |
| Agent | main、text、commentary、reviewer |
| 知识库 | 原文、脂批、核心人物、基础版本说明 |
| RAG | 混合检索思想，返回证据卡片 |
| 证据治理 | 证据包、结论声明、审校记录 |
| 验证 | 标准问题集、负面用例、上线门槛 |

## 第一版不做

1. 全量红学论文；
2. 复杂校勘；
3. 人物陪聊；
4. 自由续写；
5. 多前端；
6. 多租户复杂权限；
7. 自动学习入库；
8. 复杂可视化图谱。

## 第二版候选能力

第二版可以考虑：

1. 增加更多版本来源；
2. 增加红学研究观点库；
3. 增加人物关系可视化；
4. 增加证据链展开界面；
5. 增加引用导出；
6. 增加教学模式；
7. 增加专题问答；
8. 增加评测仪表盘。

## 第三版候选能力

第三版可以考虑：

1. 多用户协作标注；
2. 人工校勘工作台；
3. 版本差异可视化；
4. 红学观点谱系；
5. 教学课程生成；
6. 文本细读辅助；
7. 跨古典小说比较；
8. 研究报告生成。

## 迭代决策原则

进入下一版前必须确认：

1. 第一版证据链稳定；
2. 负面清单测试通过；
3. 核心问题集表现稳定；
4. 用户能理解回答分层；
5. 管理员能追踪问题；
6. 架构没有因新增能力破坏边界。

## 优先级判断

新增能力优先级按以下顺序：

1. 能提升证据可靠性的能力；
2. 能提升用户理解的能力；
3. 能提升审校和治理的能力；
4. 能提升运维和评测的能力；
5. 最后才是表达风格和娱乐性能力。

## 风险提示

如果第一版尚未稳定，不建议引入：

1. 复杂人物陪聊；
2. 自动生成红学结论；
3. 自主学习知识库；
4. 无人工校验的全网资料抓取；
5. 多 Agent 自由协商流程；
6. 用户可自定义内部 Agent。
