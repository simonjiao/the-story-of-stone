# 05 总体架构

## 架构目标

第一版总体架构要满足：

1. 用户入口简单；
2. 内部 Agent 分工清晰；
3. Gateway 非 Agent 化；
4. 知识库与回答生成解耦；
5. 证据链可追溯；
6. 审校流程不可绕过；
7. 详细设计可验证。

## 总体结构

系统分为六层：

| 层级 | 职责 |
|---|---|
| 用户入口层 | Open WebUI，一个账号或用户组只看到一个可见模型 |
| Gateway 层 | 协议适配、鉴权、会话映射、Runtime step plan、安全和审计 |
| Agent 运行层 | Runtime 内部四个 profile：main、text、commentary、reviewer |
| 知识服务层 | 原文、脂批、版本、人物、关系、事件、诗词判词、评测题库和检索服务 |
| 证据治理层 | 证据卡片、证据包、结论声明、审校记录 |
| 验证运维层 | 测试集、评测指标、日志、告警、负面清单 |

## 核心链路

标准链路：

1. Open WebUI 发起用户问题；
2. Gateway 接收并鉴权；
3. Gateway 归一化请求；
4. `honglou-main` 生成计划；
5. Gateway 根据计划和硬规则提交 Runtime step plan；
6. `honglou-text` 返回正文证据；
7. `honglou-commentary` 返回脂批和版本证据；
8. Runtime Agent/tool 形成证据包并返回 package ref；
9. `honglou-main` 生成草稿；
10. `honglou-reviewer` 审校；
11. 如未通过，主控修订；
12. Gateway 返回最终结果。

## 关键架构判断

### Open WebUI 只做入口

Open WebUI 不承担多 Agent 编排，不承担证据治理，不承担内部权限策略。它负责用户登录、聊天入口和模型选择。

### Gateway 不是第 5 个 Agent

Gateway 是确定性边界层，不生成文学解释，不自由调用工具，不替代
`honglou-main` 判断问题。目标 Runtime 接入后，Gateway 只创建受控 step plan，
不直接执行领域检索、证据包构建或 reviewer。

### Runtime 内部运行四个 Agent

四个 Agent 是受控 profile，各自有工具白名单、输入输出格式、职责边界和审校约束。

### 知识库不直接暴露给 Agent

Agent 通过受控工具访问知识库。工具返回结构化证据卡片，不返回无限制原始数据。

### 审校是硬流程

所有复杂回答必须经过 reviewer。用户不能通过提示词跳过审校。

## 架构冻结条件

总体架构可以冻结的条件：

1. 六层职责被接受；
2. Gateway 非 Agent 化被接受；
3. 四个 Agent 的边界被接受；
4. Open WebUI 单模型入口被接受；
5. 证据包与审校链路被接受；
6. 第一版不做项被接受。

## 知识服务层

知识服务层不应只是一个检索接口，而应包括以下能力：

1. 原文证据管理；
2. 脂批证据管理；
3. 版本边界管理；
4. 人物与别名管理；
5. 人物关系与事件链管理；
6. 诗词判词曲文专题管理；
7. 字形读音索引；
8. 证据卡片规范化；
9. 证据包构造；
10. 评测题库管理。

架构上坚持：RAG 层提供证据，Agent 层组织回答，reviewer 层审校边界，Gateway 层强制流程。RAG 层不得直接生成面向用户的最终回答。
