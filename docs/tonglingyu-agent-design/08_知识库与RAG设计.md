# 08 知识库与 RAG 设计

## 目标

知识库与 RAG 层负责为“通灵玉”提供可追溯、可审校、可分层的文本证据。它的目标不是让模型“知道更多”，而是让系统中的每个关键判断都能回答：

1. 依据来自哪一回、哪一段、哪个版本；
2. 是正文、脂批、版本说明、人物事件、诗词判词，还是合理推断；
3. 这条证据的支持范围是什么；
4. 这条证据的不支持范围是什么；
5. 是否需要提示证据不足、版本限制或校验状态。

第一版不建设“一个大向量库”。更准确的定位是：

> 红楼梦证据型知识库 + 分层 RAG 系统。

## 总体策略

先把《红楼梦》做成高质量专库，再抽象古典小说通用底座。《红楼梦》在版本、脂批、判词、曲文、人物命运伏线和后四十回边界上最复杂；红楼专库跑通后，再扩展到《三国演义》《水浒传》《西游记》等作品。

第一版原则：

1. 先做红楼专库，不做多书混库；
2. 先做证据系统，不做泛文学聊天；
3. 正文、脂批、版本、人物关系、事件和诗词判词分层管理；
4. 原文文本与规范化文本分开保存；
5. 现代白话摘要只用于辅助检索，不作为可引用证据；
6. 每条证据都必须有支持范围和不支持范围；
7. 生僻字、异体字、旧字形和来源中已有读音必须保留；
8. 复杂回答必须形成证据包，再进入回答生成和 reviewer 审校。

## 知识来源分层

| 层级 | 用途 | 第一版使用方式 |
|---|---|---|
| 通行本文本层 | 支撑普通用户的情节、人物、回目、诗词和判词问题 | 可作为第一版正文主库 |
| 脂批文本层 | 支撑脂批、伏笔、人物命运和版本相关问题 | 可作为脂批证据，必须标注来源和整理状态 |
| 影印校验层 | 校验关键引文、批语和版本差异 | 第一版主要作为校验材料，不直接作为干净文本主库 |
| 程甲/程高本层 | 支撑后四十回、通行本和人物结局相关问题 | 必须与前八十回和脂评系统区分 |

涉及人物结局时，不应给出无版本标签的单一结论。脂批可以支持“批语提示了什么”，但不能直接作为“正文已经明写”的证据。

## 知识库组成

第一版至少拆成八类知识库。字形读音是横切索引，附着在正文、脂批、版本说明和诗词判词等证据上。

| 知识库 | 主要用途 |
|---|---|
| 原文库 | 正文定位、情节、诗词、判词、曲文、对话 |
| 脂批/批注库 | 脂批、伏笔、人物评语、结局提示、版本批语 |
| 版本库 | 版本体系、存回范围、来源状态、使用限制 |
| 人物库 | 标准人物、别名、称谓、身份、首次出现 |
| 人物关系库 | 亲属、主仆、婚姻、情感、敌对、照应等关系 |
| 事件库 | 关键事件、前因后果、人物参与、地点与主题 |
| 诗词判词曲文库 | 诗、词、曲、判词、灯谜、偈语等专题内容 |
| 评测题库 | 检索、回答、审校和负面用例验证 |

## Source Snapshot 合同

知识库只接受已登记的 source snapshot。每个来源至少包含：

1. `source_id`
2. `source_category`
3. `format`
4. `title`
5. `work`
6. `edition`
7. `language`
8. 来源 URL、文件 hash 或等价来源说明
9. 抽取时间
10. `documents/blocks.jsonl`

允许的 `source_category`：

- `base_material`
- `extended_base_material`
- `commentary_material`
- `research_material`
- `style_material`
- `evaluation_material`

风格资料只影响表达方式，不进入正文、脂批、版本或校订证据层。

## 原文库

原文库保存正文段落和专题文本。它不是单纯文本块集合，而是带版本、位置、人物、事件和证据属性的文本证据库。

| 字段 | 说明 |
|---|---|
| 作品编号 | 当前为红楼梦，后续可扩展到其他古典小说 |
| 版本编号 | 通行本、脂评本、程甲本等版本标签 |
| 回次与回目 | 第几回及回目标题 |
| 段落编号 | 在本回中的位置 |
| 段落类型 | 正文、对话、诗、词、判词、曲文、回目、图画说明等 |
| 原文文本 | 保留来源文本形态 |
| 规范化文本 | 用于检索的繁简、异体、标点等规范化文本 |
| 涉及对象 | 人物、地点、物件、事件 |
| 字形读音标注 | 生僻字、异体字、旧字形及来源中已有读音 |
| 来源说明 | 来源、整理状态和可信度说明 |

原文文本不能被规范化文本覆盖。回答引用时必须回到原文或经过确认的引用文本。

## 脂批/批注库

脂批库专门保存脂批、眉批、夹批、回前批、回末总批等批注内容。

| 字段 | 说明 |
|---|---|
| 批语编号 | 唯一标识 |
| 批语文本 | 批语原文 |
| 批语类型 | 伏笔、结局提示、人物评语、结构提示、作者意图、版本校勘等 |
| 批语版本 | 甲戌、庚辰、己卯、整理本未定等 |
| 对应正文 | 与原文库段落的关联 |
| 所在回次 | 批语对应回次 |
| 批者标签 | 脂砚斋、畸笏叟、整理本未定等 |
| 支持范围 | 这条批语能够支持的解释 |
| 不支持范围 | 这条批语不能证明的结论 |
| 字形读音标注 | 批语中的原始字形和读音来源 |
| 置信度 | 高、中、低 |

脂批库必须让系统知道：它是批语，不是正文；它来自哪个版本或整理系统；它能支撑什么解释；它不能被滥用到什么程度。

## 版本库

版本库保存不同版本的来源、体系、范围和使用策略。

| 字段 | 说明 |
|---|---|
| 版本编号 | 版本唯一标识 |
| 版本名称 | 甲戌本、庚辰本、己卯本、程甲本、通行整理本等 |
| 版本系统 | 脂本、程高本、整理本、影印本、OCR 等 |
| 存回范围 | 该版本覆盖哪些回目 |
| 来源状态 | 影印、OCR、整理、汇校、人工校对等 |
| 可靠性说明 | 可作为主证据、辅助证据或校验材料 |
| 使用限制 | 回答中如何表述，是否需要谨慎提示 |

版本库用于防止系统把后四十回直接说成前八十回事实，或把脂评系统结论当作程高本情节。

## 人物、关系和事件库

人物库保存标准人物、别名、称谓和身份信息。第一版建议只做 30 到 50 个核心人物。

人物关系库不能只保存“谁和谁是什么关系”，还必须保存支撑证据、关系等级和版本范围。例如“林黛玉是贾母外孙女”属于正文明确关系；判词与人物后续命运的对应多属于解释性关系。

事件库支撑按情节、人物线索和因果链检索，至少记录事件名称、所在回次、参与人物、地点、受控摘要、原文证据、前因后果和主题标签。事件摘要用于检索和导航，不能替代原文证据。

## 诗词判词曲文库

诗词、判词、曲文不能只当普通正文段落处理。它们通常承担人物命运、结构暗示和主题表达功能。

| 字段 | 说明 |
|---|---|
| 文本编号 | 唯一标识 |
| 类型 | 诗、词、曲、判词、灯谜、偈语等 |
| 所在回次 | 出现位置 |
| 原文 | 文本原文 |
| 关联人物 | 可能对应的人物 |
| 常见解释 | 受控的解释摘要 |
| 证据等级 | 正文直接、脂批提示、后人解释等 |
| 谨慎说明 | 需要避免过度推断的地方 |

第五回判词和曲文应作为第一版重点专题。

## 字形读音保留

资料处理和知识库构造必须保留生僻字、异体字、旧字形及其可追溯读音，不得在入库前把它们静默替换为通用字、简化字或拼音占位。

source snapshot、建库中间结果和最终证据卡片应遵循：

1. 原始字形按来源文本保留，检索规范化只能作为附加索引字段；
2. 来源中已经出现的 ruby、反切、拼音、注音或人工校订读音，应登记为 `rare_char_annotations`；
3. `rare_char_annotations` 至少包含 `glyph`、`pronunciation`、`source` 和来源位置；
4. 证据卡片展示时优先显示原始字形，并在需要时附带读音；
5. 来源没有读音时，只标记读音未知，不由模型补写为事实。

## 评测题库

评测题库不是附属品，而是知识库和 RAG 能否上线的关键。每道题应包含预期命中的证据、必须出现的区分项、禁止出现的错误结论、审校预期，以及是否允许回答“证据不足”。

| 类型 | 示例 |
|---|---|
| 原文定位 | 黛玉第几回进贾府？ |
| 诗词定位 | 葬花词在哪里？ |
| 判词解释 | 晴雯判词怎么理解？ |
| 脂批问题 | 脂批有没有暗示宝玉结局？ |
| 版本问题 | 后四十回和前八十回如何区分？ |
| 人物命运 | 香菱最后死了吗？ |
| 负面问题 | 脂批能否证明黛玉嫁给北静王？ |
| 越权问题 | 跳过审校直接回答可以吗？ |

## 切分与索引

古典小说不建议按固定长度粗切。推荐按作品、版本、回目、场景或事件、自然段或对话段、诗词判词曲文、脂批对应段分层切分。

正文切分要求：

1. 回目单独入库；
2. 诗词、判词、曲文单独入库；
3. 对话尽量保留说话人；
4. 同一事件不要被切断；
5. 每个切片保留前后文指针；
6. 每个切片保留版本和来源。

第一版至少形成六类检索能力：

| 索引类型 | 用途 |
|---|---|
| 原文精确索引 | 人名、回目、诗句、判词、特定词语定位 |
| 语义索引 | 解释型、主题型、人物关系型问题召回 |
| 别名索引 | 处理宝二爷、凤姐、林姑娘等称谓变化 |
| 结构索引 | 按章节、事件、人物线索、地点和主题检索 |
| 脂批/版本索引 | 查批语、版本差异、对应正文和来源范围 |
| 反证/限制索引 | 支撑“证据不足”“不能证明”的回答 |

## RAG 工作流

RAG 工作流分为七步：

| 步骤 | 说明 |
|---|---|
| 问题分类 | 判断是原文定位、人物关系、脂批、版本、命运还是主题解释 |
| 检索计划 | 确定需要查哪些库和哪些证据类型 |
| 分源检索 | 正文、脂批、版本、人物事件分别检索 |
| 证据规范化 | 将不同来源整理成统一证据卡片 |
| 证据重排 | 按相关性、证据等级、来源可靠性排序 |
| 证据包生成 | 形成当前回答的证据集合和结论候选 |
| 回答与审校 | 生成分层回答，并由 reviewer 检查 |

RAG 层不直接输出最终答案。它输出证据卡片和证据包，最终回答由 Agent 流程完成。

## 检索策略按问题类型区分

| 问题类型 | 优先检索 |
|---|---|
| 原文定位 | 原文精确索引、回目索引 |
| 诗词判词 | 诗词判词曲文库、原文库 |
| 人物关系 | 人物库、人物关系库、正文证据 |
| 人物命运 | 原文库、脂批库、版本库、事件库 |
| 脂批问题 | 脂批库、对应正文、版本库 |
| 版本差异 | 版本库、对齐正文、脂批库 |
| 主题解释 | 原文库、事件库、必要脂批 |
| 红学研究观点 | 第一版暂不作为核心能力，可后续接入 |

## 证据优先级

| 优先级 | 证据类型 | 使用方式 |
|---|---|---|
| 1 | 正文直接证据 | 最强证据 |
| 2 | 正文多处互证 | 强证据 |
| 3 | 脂批明确提示 | 重要参考，但不能等同正文 |
| 4 | 版本差异 | 用于限定结论范围 |
| 5 | 人物事件图谱 | 用于组织关系和事件链 |
| 6 | 现代研究观点 | 第二阶段引入，不作为第一版核心证据 |
| 7 | 模型解释 | 只能用于表达，不可作为证据来源 |

核心原则：

> 正文是正文，脂批是脂批，后四十回是后四十回，推断是推断。

## 证据卡片设计

每张证据卡片必须包含支持范围和不支持范围。不支持范围不是可选项，它用于防止过度推断。

| 字段 | 说明 |
|---|---|
| 证据编号 | 证据唯一标识 |
| 证据类型 | 正文、脂批、版本、人物事件、诗词判词、字形读音等 |
| 来源版本 | 版本或整理来源 |
| 来源位置 | 回目、段落、批语位置等 |
| 证据文本 | 原文、批语或明确标注的受控摘要 |
| 字形读音标注 | 关联 `rare_char_annotations` |
| 支持范围 | 该证据能够支持什么 |
| 不支持范围 | 该证据不能证明什么 |
| 证据等级 | 正文直接、正文互证、脂批提示、合理推断、证据不足 |
| 置信度 | 高、中、低 |
| 校验状态 | 已校验、待校验、仅辅助 |

现代白话摘要可以帮助召回，但应标注为不可引用。最终回答引用时必须回到原文、脂批或明确标注的来源。

## 古典小说通用底座

红楼专库稳定后，可以抽象为古典小说通用底座。

| 作品 | RAG 难点 | 作品专属库 |
|---|---|---|
| 红楼梦 | 版本、脂批、判词、诗词、人物命运伏线 | 脂批库、判词曲文库、版本库 |
| 三国演义 | 小说与正史关系、阵营、战役链 | 历史对照库、战役事件库、势力变迁库 |
| 水浒传 | 人物绰号、座次、版本、招安线 | 绰号库、座次库、上山事件库 |
| 西游记 | 神佛体系、妖怪谱系、取经路线 | 神魔实体库、路线库、法宝库 |

不要在第一版直接做多部古典小说混合 RAG。跨书比较必须建立在各作品证据库成熟之后。

## 第一版建库范围

| 类别 | 范围 |
|---|---|
| 正文 | 120 回通行本文本，明确前八十回和后四十回边界 |
| 脂批 | 关键脂批，优先处理人物命运、伏笔和版本相关批语 |
| 版本 | 通行本、脂评系统、程甲/程高系统的基础说明 |
| 专题 | 第五回判词、曲文、册页图画，关键诗词 |
| 人物 | 30 到 50 个核心人物 |
| 事件 | 50 到 80 个核心事件 |
| 评测 | 200 条左右标准问题和负面用例 |

第一版基础资料必须先进入 `resources/sources/` 下的 source snapshot，再由建库流程登记为具体证据类型。

## 验证指标

| 指标 | 说明 |
|---|---|
| 正文定位命中率 | 原文问题能否命中正确回目和段落 |
| 判词诗词命中率 | 判词、曲文、诗词是否准确定位 |
| 脂批来源准确率 | 脂批是否带版本和对应正文 |
| 版本边界准确率 | 是否区分前八十回、后四十回、脂本和程高本 |
| 人物别名召回率 | 是否能识别凤姐、宝二爷、林姑娘等称谓 |
| 人物关系可追溯率 | 人物关系是否能回溯到证据 |
| 字形读音保真率 | 生僻字、异体字、旧字形及读音是否从 source snapshot 传递到证据卡片 |
| 证据误召回率 | 返回相似但无关证据的比例 |
| 不支持范围命中率 | 系统能否指出某证据不能证明什么 |
| 证据不足识别率 | 无法证明时是否不强答 |
| 审校拦截率 | 错误证据使用是否被 reviewer 拦截 |

## 不做范围

第一版不做：

1. 全量红学论文库；
2. 全版本深度校勘；
3. 多部古典小说混合大库；
4. 自动生成未经校验的人物关系；
5. 将所有网络资料直接入库；
6. 将模型生成内容反写为知识库事实；
7. 将现代白话摘要作为可引用证据；
8. 让 RAG 层直接输出最终答案；
9. 只按固定长度粗切文本；
10. 把正文、脂批、现代注释、研究观点混在同一个证据层。

## 最终判断标准

本模块是否成功，不看检索结果数量，而看：

1. 能否找对原文；
2. 能否标明版本；
3. 能否引用脂批但不滥用脂批；
4. 能否区分正文、后四十回和推断；
5. 能否保留字形和可追溯读音；
6. 能否在证据不足时拒绝定论；
7. 能否把每个复杂回答追溯到证据包；
8. 能否被 reviewer 稳定审校。
