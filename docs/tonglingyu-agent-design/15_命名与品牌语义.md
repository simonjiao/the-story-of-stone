# 15 命名与品牌语义

## 推荐名称

第一版推荐产品名：

> 通灵玉

推荐完整表达：

> 通灵玉：红楼文本证据与脂批问答系统

推荐用户可见入口名称：

> 通灵玉

推荐一句话说明：

> 一块能查原文、引脂批、讲版本、辨推断的“通灵之玉”。

## 为什么不用“红楼研究助手”

“红楼研究助手”准确但偏功能化，像一个工具说明，不像一个有《红楼梦》内部语义的产品名称。它适合作为副标题，不适合作为主品牌。

## 为什么推荐“通灵玉”

“通灵玉”有三个优势：

1. 来自原著核心意象，用户一眼能进入《红楼梦》的语境；
2. 与“记录、见闻、转述、追溯”的产品机制天然贴合；
3. 比“红楼梦研究助手”更短、更有记忆点，也更适合出现在 Open WebUI 的模型列表里。

## 叙事依据的稳妥表述

产品文案中建议使用稳妥表述：

> 《红楼梦》的叙事框架与“石头记”有关，故事被设定为石头亲历、记录并经抄录流传。“通灵玉”这一名称借用了这种“见证者”和“记录者”的意象。

不建议写成过强表述：

> 全书所有内容都是通灵玉亲眼看到、亲耳听到并写在玉上的。

原因是，原著框架更准确地说是“石头/通灵宝玉”的经历、记述、被抄录与流传，而不是现代意义上严格的第一视角摄录。产品名可以借用这个意象，但说明文字应保持学术谨慎。

## 名称分层

| 层级 | 推荐命名 | 说明 |
|---|---|---|
| 产品主名 | 通灵玉 | 用户看到的名字 |
| 产品副标题 | 红楼文本证据与脂批问答系统 | 用于解释能力 |
| Open WebUI 显示名 | 通灵玉 | 简短、好记 |
| 内部项目名 | Honglou Hermes Agent | 技术项目名，可保留 |
| 内部 Agent 名 | main、text、commentary、reviewer | 不建议改成文学化名称，便于工程协作 |

## 可选备选名称

| 名称 | 风格 | 评价 |
|---|---|---|
| 通灵玉 | 原著核心意象 | 最推荐，短、准、有记忆点 |
| 石头记问 | 偏文本研究 | 学术感强，但不如通灵玉亲切 |
| 石上书 | 偏诗意 | 好听，但功能指向稍弱 |
| 大观笺 | 偏文雅 | 适合内容产品，不如通灵玉贴合系统机制 |
| 脂砚灯 | 偏脂批研究 | 适合脂批专题，不适合作为全系统名 |
| 太虚镜 | 偏意象化 | 有审视、观照意味，但可能显得玄 |
| 红楼玉牒 | 偏档案/谱系 | 适合人物关系图谱模块，不适合作为总名 |

## 最终建议

主品牌使用：

> 通灵玉

完整介绍使用：

> 通灵玉是一套面向《红楼梦》的文本证据与脂批问答系统。它不扮演人物，不自由续写，而是基于原文、脂批、版本和人物事件证据，给出可追溯、可审校、可分层的回答。

## 对文档的影响

所有面向用户和产品评审的表述应从“红楼梦研究助手”改为“通灵玉”。

内部技术组件无需文学化改名：

1. Gateway 仍称 Gateway；
2. 四个 Agent 仍称 main、text、commentary、reviewer；
3. 证据包、审校、知识库、RAG 等术语保持不变。

这样可以同时兼顾产品记忆点和工程可理解性。
