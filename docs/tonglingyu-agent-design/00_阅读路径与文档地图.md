# 00 阅读路径与文档地图

## 目标

本套文档用于指导“红楼梦研究型 Hermes Agent”第一版落地。文档采用逐步披露原则：先确认产品边界，再确认架构责任，最后进入可验证的详细设计。

## 逐步披露层级

| 层级 | 面向对象 | 目标 | 对应文档 |
|---|---|---|---|
| L0 总览层 | 决策者、产品负责人、架构负责人 | 快速确认做什么、不做什么、如何验收 | 00、01、02、03 |
| L1 产品层 | 产品经理、领域专家、业务负责人 | 明确用户、场景、边界、回答风格和证据原则 | 01、02、04 |
| L2 架构层 | 架构师、技术负责人 | 明确系统构成、职责分离、Agent 编排和 Gateway 定位 | 05、06、07、08 |
| L3 详细设计层 | 后端、算法、数据、测试 | 明确接口语义、内部协作、数据对象、权限审计 | 09、10、11 |
| L4 验证层 | 测试、验收、运维、治理人员 | 用清单、指标和反例验证系统是否可上线 | 12、13、14 |
| L5 实施落地层 | 工程负责人、实现人员 | 对齐当前仓库差距、第一阶段实施顺序和后续演进边界 | 16、17、18、19 |

## 当前设计重点

本轮知识库与 RAG 设计的核心变化是把 RAG 定位为证据系统，而不是普通检索模块。评审时优先看：

1. `08_知识库与RAG设计.md`：证据型知识库、分层 RAG、八类知识库和证据卡片；
2. `04_概念模型与证据分层.md`：证据卡片、证据包、支持范围和不支持范围；
3. `12_验证方案与验收标准.md`：RAG 指标、负面用例和上线门槛；
4. `17_完整知识库与风格扩展规划.md`：完整知识库、研究观点层和多风格扩展的后续边界；
5. `18_第一版实施细化计划.md`：按阶段列出产物、验收和需要澄清的问题；
6. `19_第一批资料来源登记.md`：登记第一批 Wikisource 资料、版本边界和暂缓来源。

## 文档边界

本套文档不绑定具体编程语言、框架、数据库、向量库、部署工具或 SDK。接口文档只描述能力、输入输出语义、状态、错误类别和验收方法，不描述具体实现代码。

## 版本治理

每份文档建议维护以下元信息：

| 字段 | 说明 |
|---|---|
| 文档编号 | 与文件编号一致 |
| 文档责任人 | 对内容正确性负责的人 |
| 适用版本 | 第一版、第二版等 |
| 决策状态 | 草案、待评审、已冻结、废弃 |
| 变更记录 | 记录关键变更原因 |
| 验证方式 | 该文档如何被验证 |

## 冻结顺序

建议按以下顺序冻结：

1. 产品定位与负面清单；
2. 用户场景与回答边界；
3. 总体架构和 Agent 分工；
4. Gateway 定位；
5. 外部接口与内部接口；
6. 验证方案和上线门槛。

原因是：产品边界不冻结，后续接口会反复变；Agent 分工不冻结，详细设计不可验证。
