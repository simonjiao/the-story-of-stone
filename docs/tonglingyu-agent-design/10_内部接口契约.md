# 10 内部接口契约

## 目标

内部接口用于 Gateway、Agent Runtime、Runtime profile、知识服务、证据服务和
审校能力之间协作。本文档只定义语义契约，不绑定具体实现方式。

## 内部接口分组

| 接口组 | 调用方 | 被调用方 | 目的 |
| --- | --- | --- | --- |
| Agent 调用接口 | Gateway | Agent Runtime Adapter | 调用指定 profile 或 step plan 执行任务 |
| 上下文投影接口 | Gateway | Context Governance | 为不同 consumer 生成可见 context projection |
| 正文检索接口 | `honglou-text` 或 read-only tool | 知识服务 | 获取正文证据 |
| 脂批检索接口 | `honglou-commentary` 或 read-only tool | 知识服务 | 获取脂批证据 |
| 版本比较接口 | `honglou-commentary` 或 read-only tool | 知识服务 | 获取版本差异 |
| 人物事件查询接口 | Runtime profile 或 read-only tool | 知识服务 | 获取人物关系和事件链 |
| 证据包接口 | Runtime profile/tool，Gateway admin proxy | 证据服务 | 创建、读取、回放证据包 |
| 审校接口 | Runtime step plan | `honglou-reviewer` profile | 审查草稿 |
| 审计记录接口 | Gateway 与 Agent Runtime | 审计服务 | 写入追踪和调用日志 |

## Agent 调用接口

### 目的

让 Gateway 以受控方式调用 Agent Runtime，执行指定 consumer/profile 或已授权的
Runtime step plan。

### 输入语义

1. 追踪编号；
2. 会话编号；
3. 目标 consumer/profile；
4. 任务名称；
5. 任务输入；
6. 工具权限约束；
7. 超时和重试策略；
8. 输出格式要求；
9. schema version；
10. profile step 依赖和 output_ref；
11. `context_pack_ref`；
12. `context_pack_schema_version`；
13. `context_pack_digest`；
14. `context_projection_ref`；
15. `context_projection_schema_version`；
16. `context_projection_digest`；
17. `consumer_type`；
18. `consumer_name`；
19. `runtime_adapter`；
20. `tool_policy_digest`；
21. `output_contract_digest`。

### 输出语义

1. 任务状态；
2. 结构化结果；
3. 调用摘要；
4. 错误类别；
5. 运行时长；
6. 是否符合 schema；
7. RuntimeStreamEvent 或最终 RuntimeOutput；
8. evidence/package ref。

### 验收标准

1. 用户不能直接构造该接口请求；
2. consumer/profile 只能执行被授权任务；
3. 输出不符合 schema 时必须拦截；
4. 超时必须被 Gateway 处理；
5. Gateway 不能通过请求字段给 Runtime profile 扩权；
6. 内部 profile 不能作为 Open WebUI 可见模型暴露；
7. `context_pack_ref` 或 `context_projection_ref` 缺失、schema 不支持、digest
   不匹配、consumer 不匹配或 Runtime Adapter 不匹配时必须 fail-closed；
8. Runtime step plan 的 allowed tools 必须与 `tool_policy_digest` 绑定；
9. 当前 Phase 2 只接受 `consumer_type=runtime_profile` 和已登记 `honglou-*`
   consumer；未知 consumer 与 `external_agent` 必须 fail-closed。

## 上下文投影接口

### 目的

为不同 Runtime consumer 生成不同可见上下文，避免信息过载和权限混乱。
详细对象、scope、journal 和 memory 边界见
`26_Scoped_Context与受控Memory设计.md`。

### 输入语义

1. 用户问题；
2. 会话摘要；
3. consumer 类型；
4. 任务类型；
5. 策略约束；
6. `interaction_context_id`；
7. scope binding；
8. memory read policy；
9. `consumer_name`；
10. `runtime_adapter`。

### 输出语义

1. 该 consumer 可见的问题内容；
2. 必要的上下文摘要；
3. 禁止使用的上下文范围；
4. 可调用 read-only tools 列表；
5. 输出约束；
6. `context_pack_id`；
7. `context_pack_ref`；
8. context pack schema version 和 digest；
9. `context_projection_id`；
10. `context_projection_ref`；
11. context projection schema version 和 digest；
12. `consumer_type`、`consumer_name` 和 `runtime_adapter`；
13. `tool_policy_digest` 和 `output_contract_digest`；
14. session journal ref。

### 验收标准

1. text consumer 不获得完整用户历史；
2. commentary consumer 不获得无关正文库原始数据；
3. reviewer 只获得草稿和证据包；
4. main 获得必要上下文但不直接获得数据库权限；
5. 外部用户不能指定 context、scope、memory 或 tool policy；
6. memory ACL 不匹配时必须 fail-closed；
7. context pack 与 context projection 可按 trace 回放，但不向普通用户暴露；
8. Runtime profile 只能读取自己的 projection，不读取完整 context pack；
9. 未知 consumer、未知 Runtime Adapter 和 `external_agent` 在 Phase 2 必须
   fail-closed。

## 正文检索接口

### 输入语义

1. 检索问题；
2. 人物、回目、版本、段落类型等条件；
3. 返回数量；
4. 是否需要原文片段；
5. 是否需要支持范围和不支持范围。

### 输出语义

1. 正文证据卡片；
2. 回目和位置；
3. 原文片段或摘要；
4. 支持范围；
5. 不支持范围；
6. 置信度。

### 验收标准

1. 必须返回来源；
2. 不返回无来源概括；
3. 不混入脂批；
4. 未命中时返回证据不足。

## 脂批检索接口

### 输入语义

1. 检索问题；
2. 版本条件；
3. 回目或人物条件；
4. 批语类型；
5. 是否需要对应正文。

### 输出语义

1. 脂批证据卡片；
2. 批语版本；
3. 对应正文位置；
4. 批语类型；
5. 支持范围；
6. 不支持范围；
7. 置信度。

### 验收标准

1. 必须标注版本；
2. 必须说明对应位置；
3. 不得将脂批作为正文事实返回；
4. 无法定位来源时标记证据不足。

## 证据包接口

### 目的

将证据卡片、结论声明和审校记录统一管理。
目标 Runtime 接入后，证据包创建、读取和 replay 由 Runtime Agent/tool 侧执行；
Gateway 只保存 package ref 或提供 admin proxy。

### 输入语义

1. 用户问题；
2. 证据卡片；
3. 结论声明；
4. 追踪编号；
5. 生成阶段；
6. Runtime profile step 和 output_ref。

### 输出语义

1. 证据包编号；
2. 证据卡片列表；
3. 结论声明列表；
4. 证据等级；
5. 审校状态；
6. replay ref。

### 验收标准

1. 每个复杂回答都有证据包；
2. 每条关键结论能映射证据；
3. 证据包可追溯到请求；
4. 证据包不可被普通用户任意篡改；
5. replay 不依赖上游模型。

## 审校接口

### 输入语义

1. 用户问题；
2. 草稿答案；
3. 证据包；
4. 回答策略；
5. 负面清单。

### 输出语义

1. 是否通过；
2. 问题列表；
3. 严重程度；
4. 必须修订项；
5. 可保留结论；
6. 审校摘要。

### 验收标准

1. 无证据断言必须被拦截；
2. 脂批和正文混淆必须被拦截；
3. 前八十回和后四十回混淆必须被拦截；
4. 版本缺失必须至少警告；
5. 用户不能关闭审校。

## RAG 内部接口

知识服务相关内部接口应按证据类型拆分，而不是提供一个无差别的大检索接口。

| 内部能力 | 目的 |
| --- | --- |
| 原文证据检索 | 查询正文、回目、段落、诗词、判词、曲文 |
| 脂批证据检索 | 查询批语、批语类型、批语版本和对应正文 |
| 版本边界查询 | 查询版本系统、存回范围、来源状态和使用限制 |
| 人物别名解析 | 将称谓、别名、号映射到标准人物 |
| 人物关系查询 | 查询人物关系及其证据等级 |
| 事件链查询 | 查询人物经历、事件前因后果和主题标签 |
| 字形读音查询 | 查询生僻字、异体字、旧字形和可追溯读音 |
| 证据卡片规范化 | 将不同来源结果整理为统一证据卡片 |
| 证据包创建 | 汇总证据卡片和结论候选 |
| 不支持范围校验 | 检查草稿是否滥用了证据 |

所有知识服务结果必须返回证据类型、来源、支持范围、不支持范围、校验状态和置信度。不得只返回自然语言概括。
