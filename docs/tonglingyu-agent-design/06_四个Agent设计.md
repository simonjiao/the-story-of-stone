# 06 四个 Agent 设计

## 总览

第一版设置四个内部 Agent：

| Agent | 定位 | 是否对用户可见 | 是否输出最终回答 |
|---|---|---:|---:|
| `honglou-main` | 主控、计划、组织答案 | 否 | 是 |
| `honglou-text` | 正文证据检索 | 否 | 否 |
| `honglou-commentary` | 脂批与版本证据 | 否 | 否 |
| `honglou-reviewer` | 审校与质量闸门 | 否 | 否 |

目标 Runtime 接入后，四个内部 Agent 都是受 profile contract 约束的 Runtime
profile。`honglou-text` 和 `honglou-commentary` 也走 LLM profile，通过
read-only tools 获取证据，而不是 Gateway 内的确定性检索 step。完整接入计划见
`20_Runtime接入设计与实施计划.md`。

## `honglou-main`

### 职责

1. 理解用户问题；
2. 判断问题类型；
3. 生成检索计划；
4. 汇总证据；
5. 生成草稿；
6. 根据审校意见修订；
7. 输出最终回答。

### 输入

1. 用户问题；
2. 最近必要会话上下文；
3. Gateway 传入的策略约束；
4. 证据包；
5. reviewer 审校意见。

### 输出

1. 问题分类；
2. 检索计划；
3. 草稿答案；
4. 修订后答案；
5. 结论声明和证据等级。

### 禁止

1. 不直接访问知识库；
2. 不绕过 reviewer；
3. 不把无证据推断写成事实；
4. 不暴露内部配置。

## `honglou-text`

### 职责

1. 检索正文；
2. 定位回目；
3. 提取诗词、判词、曲文；
4. 返回证据卡片；
5. 标明证据的支持范围和不支持范围。

### 输入

1. 检索主题；
2. 人物、回目、版本等过滤条件；
3. 返回数量和证据类型要求。

### 输出

1. 正文证据卡片；
2. 回目、版本、位置；
3. 原文片段或摘要；
4. 支持范围和不支持范围。

### 禁止

1. 不解释脂批；
2. 不生成最终回答；
3. 不进行版本结论裁决；
4. 不使用影视或网络设定。

## `honglou-commentary`

### 职责

1. 检索脂批；
2. 定位对应正文；
3. 标注批语版本；
4. 区分批语类型；
5. 判断脂批支持范围；
6. 处理必要的版本差异。

### 输入

1. 脂批或版本相关问题；
2. 目标人物、回目、批语类型；
3. 版本过滤条件。

### 输出

1. 脂批证据卡片；
2. 对应正文位置；
3. 批语类型；
4. 版本说明；
5. 支持范围和不支持范围。

### 禁止

1. 不把脂批当正文事实；
2. 不输出最终回答；
3. 不把单条批语扩展成确定结论；
4. 不忽略版本来源。

## `honglou-reviewer`

### 职责

1. 检查草稿是否有证据；
2. 检查引文是否支持结论；
3. 检查正文、脂批、推断是否混淆；
4. 检查版本是否混淆；
5. 检查是否存在过度断言；
6. 输出通过、警告或拦截意见。

### 输入

1. 用户问题；
2. 草稿答案；
3. 证据包；
4. 回答策略。

### 输出

1. 审校结果；
2. 问题列表；
3. 严重程度；
4. 必须修订的表述；
5. 可通过的结论声明。

### 禁止

1. 不重新生成最终答案；
2. 不替代正文和脂批 Agent 大规模检索；
3. 不绕过主控 Agent 直接返回用户；
4. 不泄露内部审校规则全文给普通用户。

## Agent 之间的边界

| 问题 | 归属 |
|---|---|
| 用户问题属于哪类？ | `honglou-main` |
| 这句话在第几回？ | `honglou-text` |
| 某批语来自哪个版本？ | `honglou-commentary` |
| 这个回答是否过度推断？ | `honglou-reviewer` |
| 是否返回给用户？ | Gateway 根据 reviewer 结果决定 |

## 验证方式

每个 Agent 都应有单独测试：

1. 主控是否正确分类；
2. 正文 Agent 是否找对回目和原文；
3. 脂批 Agent 是否标明版本和批语边界；
4. 审校 Agent 是否拦截错误结论；
5. 全链路是否执行固定顺序。

## 与 RAG 的协作边界

| Agent | 与 RAG 的关系 |
|---|---|
| main | 生成检索计划，接收证据包，组织分层回答 |
| text | 通过 `tonglingyu.text.search` 调用原文、诗词判词、人物别名和事件证据能力 |
| commentary | 通过 `tonglingyu.commentary.search` 调用脂批、版本和对应正文能力 |
| reviewer | 检查证据卡片是否支持草稿结论，尤其检查不支持范围 |

`honglou-text` 返回的结果必须是正文证据卡片，不得混入脂批解释。`honglou-commentary` 返回的结果必须明确批语版本和对应正文，不得把脂批当作正文事实。`honglou-reviewer` 必须检查主控 Agent 是否违反证据的不支持范围。

## Runtime Profile Contract 摘要

完整字段以 `20_Runtime接入设计与实施计划.md` 为准。这里保留职责摘要：

1. `honglou-text`：输入用户问题、检索意图、版本/回目/人物条件和 top_k；
   输出正文证据分析、支持范围、不支持范围和 evidence refs。
2. `honglou-commentary`：输入用户问题、脂批/版本问题、版本条件和对应正文
   需求；输出脂批证据分析、对应正文、支持范围、不支持范围和 evidence refs。
3. `honglou-main`：输入用户问题、text/commentary 输出、证据包 ref 和回答
   策略；输出草稿回答、claim statements 和证据引用关系。
4. `honglou-reviewer`：输入用户问题、草稿、证据包 ref、claim statements 和
   负面清单；输出 review status、issues、severity 和 required revisions。
