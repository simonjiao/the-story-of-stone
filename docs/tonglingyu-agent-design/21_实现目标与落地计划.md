# 21 实现目标与落地计划

## 文档定位

本文用于校准通灵玉当前阶段的目标口径和后续落地顺序。它不替代
`12_验证方案与验收标准.md`、`17_完整知识库与风格扩展规划.md`、
`18_第一版实施细化计划.md` 或 `20_Runtime接入设计与实施计划.md`。

当前口径分三层：

1. 生产入口 production-ready：
   Gateway + Runtime Agent + Open WebUI 入口 + release gate。
2. 二阶段知识治理：
   召回质量、失败样本、专家复核、KB version diff 和知识修正闭环。
3. 长期演化：
   研究观点层、风格层、专家标注工作台、多 Agent 扩展和跨作品底座。

本文的主要校准是：当前可以按 R5D gate 口径声明生产入口已达到
production-ready，但不能把这句话扩大解释为完整红楼梦知识库、专家标注体系或
长期多 Agent 能力已经完成。

## 解释顺序

若本文和其他设计文档出现理解差异，按以下顺序解释：

1. 当前完成度以 `PROGRESS.md` 为准；
2. Runtime/Gateway 边界以 `20_Runtime接入设计与实施计划.md` 为准；
3. 第一版最小证据闭环以 `18_第一版实施细化计划.md` 为准；
4. 完整知识库、风格、研究观点和专家标注按
   `17_完整知识库与风格扩展规划.md` 后续演进处理。

## 一致性复盘

对照之前设计文档后，结论如下。

一致的部分：

1. 第一版主线仍是证据型 RAG 闭环：
   `source snapshot -> 知识库 -> 证据卡片 -> 证据包 -> reviewer -> 分层回答`。
2. Open WebUI 仍只暴露 `tonglingyu`，不暴露 `honglou-*` 内部 profile。
3. reviewer、证据包、trace、admin audit 和 release readiness 仍是生产入口闸门。
4. 风格、研究观点、专家标注和跨作品能力仍是后续演进，不进入当前事实层。
5. 当前优先级继续保持“通俗分析优先”；影印件和权威校注本复核作为后续 QA
   或证据校验增强，不追溯阻塞当前生产入口。

原文容易造成不一致的地方已经收敛：

1. “production-ready”只指生产入口和发布闸门，不指完整红楼梦专库完成。
2. `RetrievalQualityReport`、`retrieval_failures`、KB version diff 和专家复核流
   是二阶段知识治理目标，不是当前 R5D 完成口径。
3. 人物、关系、事件、诗词判词、terms 和 evaluation cases 是逐步补齐的
   业务数据层；空表或未专家复核数据不反向否定当前入口验收。
4. release artifact 有环境绑定和新鲜度要求，不能作为永久有效证明。

## 当前基线

当前仓库已经具备：

1. 第一批 Wikisource source snapshot 工程基线；
2. SQLite/FTS 知识库构建和本地验证；
3. `tonglingyu-runtime` 承接 source snapshot loader、KB schema、FTS、alias、
   evidence card、evidence package、reviewer、replay、audit 和 runtime store；
4. `tonglingyu-gateway` 作为 OpenAI-compatible 入口，负责鉴权、限流、会话、
   trace、SSE、模型隐藏、admin proxy、响应封装和受控 Runtime step plan；
5. 四个内部 Runtime profile：
   `honglou-text`、`honglou-commentary`、`honglou-main`、`honglou-reviewer`；
6. smoke、eval、strict Gateway gate、Open WebUI Function / Action gate、
   release readiness report 和 browser review evidence 流程。

当前不应声明已经完成：

1. 完整红楼梦专库；
2. 人物、关系、事件、诗词判词达到专家验收规模；
3. `RetrievalQualityReport`、`retrieval_failures`、KB version diff 已实现；
4. 影印件或权威校注本复核已替代当前 Wikisource source snapshot；
5. 过期 release artifact 仍能证明当前环境 production-ready。

## 产品目标

通灵玉不是普通聊天机器人，也不是单纯向量 RAG。它是面向《红楼梦》的
研究型证据助手：

1. 先找证据；
2. 再组织证据包；
3. 经 reviewer 审校；
4. 输出分层回答；
5. 复杂回答必须可追溯、可回放、可复核。

最终用户只感知 Open WebUI 中的“通灵玉”模型入口；内部运行 Gateway、Runtime、
四个 profile、证据工具、审计和评测体系。

## 当前不做

生产入口阶段不做以下事项：

1. 不让普通用户选择内部 `honglou-*` profile；
2. 不允许用户自定义工具权限、Runtime step plan 或 reviewer 开关；
3. 不把现代白话摘要作为可引用证据；
4. 不把风格资料作为正文、脂批、版本或校勘证据；
5. 不让模型生成内容自动反写为知识库事实；
6. 不把研究观点覆盖正文、脂批或版本事实；
7. 不在证据链稳定前开放自由 style prompt；
8. 不在红楼梦专库稳定前做多书混合 RAG；
9. 不把 release readiness 局部通过误判为生产完成；
10. 不把 Agent 建议等同于专家确认。

## 阶段计划

### A. 生产入口口径固化

目标：把当前状态固定为“生产入口已 production-ready，知识治理继续增强”。

产物：

1. 本文档；
2. `PROGRESS.md` 中的当前完成度摘要；
3. release artifact 索引；
4. 已知风险和下一阶段 checklist。

当前 artifact 索引：

1. 当前完成度事实源：`docs/tonglingyu-agent-design/PROGRESS.md`；
2. Runtime/Gateway R5D checklist：
   `docs/tonglingyu-agent-design/20_Runtime接入设计与实施计划.md`；
3. 远端 production report 记录路径：
   `/home/simon/hermes-home-deploy/tonglingyu-release-readiness-production.json`；
4. 远端 browser review evidence 记录路径：
   `/home/simon/hermes-home-deploy/openwebui-browser-review/openwebui-browser-review.json`；
5. 本地 eval 报告默认路径：`data/tonglingyu/reports/`。

这些路径是复核入口，不是永久有效证明。声明当前环境仍 production-ready 前，
必须重新运行 release readiness，或确认保存后的报告仍在有效期内且通过 validator。

### B. 生产入口持续加固

目标：保持 Gateway/Runtime/Open WebUI/release gate 长期为绿。

必须保持：

1. Open WebUI 普通用户只看到 `tonglingyu`；
2. Gateway/admin key 隔离；
3. rate limit、body limit、safe error、internal field rejection；
4. streaming 和 non-streaming 响应边界一致；
5. Gateway 不回流 source snapshot、FTS、evidence package、reviewer 领域逻辑；
6. Runtime store 承接 domain schema、tools、package、reviewer、audit；
7. backup、retention、prune dry-run 和 restore 演练可执行；
8. release readiness 是生产声明前置条件；
9. 普通响应不泄露 runtime/admin trace 内部字段。

验收：

1. `/v1/models` 只暴露通灵玉；
2. 公共请求拒绝内部 runtime/admin 控制字段；
3. 公共响应不泄露内部 runtime/admin trace 字段；
4. 复杂回答有 trace id、package id 和 reviewer 记录；
5. package replay 可用；
6. admin trace 中 runtime summary 与 audit event 一致；
7. release readiness report 可被 saved-report validator 复核；
8. browser review evidence 绑定 public Open WebUI URL、review ref 和证据文件。

阶段 B 不要求完整红楼梦专库、专家标注工作台或研究观点层完成。

### C. 多角色评测闭环

目标：让普通用户、管理员、专家和 Agent 分工参与 eval。

普通用户负责：

1. 入口是否只看到通灵玉；
2. 回答是否清楚区分正文、脂批、版本和推断；
3. 证据不足时是否明确说明；
4. streaming 聊天体验是否正常；
5. 是否有内部字段、trace 细节或错误栈泄露。

管理员负责：

1. health、models、metrics、Prometheus；
2. rate limit、body limit、admin key isolation；
3. trace、session、package、reviewer、runtime summary；
4. package replay；
5. release readiness report；
6. browser review evidence。

专家负责：

1. source snapshot 来源适用性；
2. 版本边界和脂批对应正文；
3. expected evidence 和 forbidden conclusion；
4. 人物、关系、事件、诗词判词 seed 是否过度推断；
5. rare char、异体字、旧字形是否被正确保留；
6. retrieval failure 根因和修正建议是否采纳。

Agent 和自动化负责：

1. schema、hash、block id、source id 校验；
2. alias、exact term、required evidence type 覆盖统计；
3. expected evidence hit@k 和 forbidden conclusion 文本检查；
4. duplicate / near-duplicate evidence 检查；
5. source / edition diversity 统计；
6. retrieval failure 聚类和诊断草案；
7. eval report、KB version diff 和 admin trace 摘要。

Agent 不能最终确认版本边界、expected evidence id、新 source 或事实层修正。

### D. 召回质量治理

目标：把当前已有 evidence card、evidence package、eval、reviewer 和 admin trace
补齐为召回质量治理闭环。该阶段对应
`17_完整知识库与风格扩展规划.md` 的 V1.5，不表示当前 Gateway/Runtime 接入未完成。

新增对象：

1. `RetrievalQualityReport`；
2. `retrieval_failures`；
3. eval 增强指标；
4. admin trace 中的 retrieval quality summary；
5. retrieval failure 的人工处理状态。

`RetrievalQualityReport` 至少覆盖：

1. trace id、tool、question；
2. query terms、protected terms、expanded aliases；
3. required / actual / missing evidence types；
4. candidate count、selected count、candidate channel；
5. source diversity、edition diversity；
6. exact term coverage；
7. expected evidence hit；
8. quality status、issues、follow-up。

`retrieval_failures` 至少覆盖：

1. failure id、trace id、package id、question、kb version；
2. failure type、query terms、required / actual evidence types；
3. expected / selected evidence ids；
4. quality issues、agent diagnosis、proposed fix；
5. human review status、reviewer、note、created/resolved time。

状态流转：

```text
recorded -> clustered -> proposed -> human_reviewed
  -> accepted/rejected -> fixed -> verified
```

eval 增强指标：

1. expected_evidence_hit@1 / @3 / @8；
2. required_type_coverage；
3. exact_term_coverage；
4. alias_expansion_hit；
5. source_diversity；
6. edition_diversity；
7. forbidden_conclusion_avoided；
8. reviewer_status_matched；
9. evidence_package_replay_passed；
10. public_response_boundary_passed；
11. admin_trace_available；
12. ordinary_user_visibility_passed。

本轮 RQA production-ready 实现中，以上对象不是可选增强。只有当 report、
failure、eval、admin trace 和 release quality gate 全部实现并通过后，才允许声明
RQA production-ready。实现期间，现有 production 入口仍以 runtime config、
model upstream、strict Gateway、Open WebUI Function、Gateway Admin Action 和
browser review gate 为准，但不得用这些既有 gate 替代 RQA 质量 gate。

本轮也不能只完成 RQA-1 到 RQA-5 后停止。RQA-6 到 RQA-12 必须落到可持久化的
governance task、人工状态流转、KB version diff 和 release blocker；否则只能算
召回质量观测，不算 production-ready 治理闭环。

生产级实现还必须覆盖运行期语义：schema migration、RQA 写入失败、重复请求、
并发状态更新、backup/restore、retention/prune 和真实 live release。只在本地测试库
或 fixture gate 中通过，不能作为 RQA production-ready。

RQA 也必须继承 `11_权限审计与安全治理.md` 的审计边界：默认不得保存完整用户
隐私文本，query、question、failure detail、metrics 和 release report 都必须脱敏、
限长、低基数，并提供稳定 admin API contract。诊断便利不能高于隐私和可运维性。

最后，RQA production-ready 还必须具备发布值守能力。release gate 通过本身不够；
必须有 operator runbook、告警条件、回滚路径和 post-release 监控证据。没有这些，
只能声明实现完成或 release candidate，不能声明 production-ready。

事故模式也不能成为绕过口。若临时关闭 RQA、跳过持久化、进入 degraded mode 或
缺少容量/审计历史证据，只能产生 non-production 状态；不能一边关闭治理链路，
一边宣称 production-ready。

RQA admin 查询本身也属于敏感操作。read/list/update 都必须鉴权、限流、脱敏审计
和防枚举；不能只审计状态更新，而让管理员读取、过滤、翻页或失败访问不留痕。

### E. 二阶段知识治理

目标：把知识库从“可用”推进到“可治理、可修正、可版本化、可专家复核”。

当前优先级仍是“通俗分析优先”。影印件、权威校注本和更强校勘材料作为后续证据
校验或发布 QA 闸门补充，不反向阻塞当前 production-ready 入口。

主要工作：

1. 完善 `kb_version`：输入 hash、schema、build report、eval report、
   release report、low confidence count、retrieval failure count；
2. 建立 source snapshot 入库报告；
3. 建立低置信证据清单；
4. 建立脂批对应正文复核清单；
5. 建立版本说明复核清单；
6. 建立人物、关系、事件 seed 的专家复核流；
7. 建立 expected evidence 和 forbidden conclusion 的专家复核流；
8. 建立 retrieval failure 修正闭环；
9. 建立 KB rebuild 后的 eval 对比报告。

知识库修正状态流转：

```text
candidate -> pending_expert_review -> accepted/rejected
  -> staged -> rebuilt -> evaluated -> released
```

Agent 可以生成 candidate，但必须由人工或专家确认后才能进入 `accepted`。

### F. 长期演化基础

长期目标不要求在当前 production-ready 入口一次完成，但必须遵守以下红线：

1. 新 profile 必须有 contract；
2. 新 tool 必须有 tool policy；
3. 新知识层必须有 source 和 review 状态；
4. 研究观点层必须与事实层分离；
5. 风格层必须在 reviewer 前受控应用；
6. 标注必须经过状态流转；
7. 用户反馈不能直接写入事实层；
8. 多 Agent 协作必须进入 trace 和 audit；
9. profile 之间不能共享未授权上下文；
10. 生产入口仍保持单模型可见。

候选 profile 仅作为后续演化，不对普通用户可见：

1. `honglou-retrieval-auditor`；
2. `honglou-knowledge-curator`；
3. `honglou-version-specialist`；
4. `honglou-commentary-linker`；
5. `honglou-style-renderer`；
6. `honglou-research-opinion`；
7. `honglou-ops-observer`。

## 验收标准

生产入口验收：

1. 普通用户只能通过 Open WebUI 的“通灵玉”入口使用；
2. 普通用户不能指定内部 profile；
3. 普通用户不能关闭 reviewer；
4. 公共请求不能伪造 runtime/admin 控制字段；
5. 公共响应不能泄露内部 runtime/admin trace 字段；
6. 每次复杂回答都有 trace id 和 evidence package ref；
7. 每个关键 claim 能映射到 evidence card；
8. reviewer 能拦截无证据断言、脂批误用、版本混淆、影视污染和 prompt injection；
9. 管理员能按 trace/package/session 复盘；
10. package replay 可用；
11. eval 报告可落盘，并能回溯到本次 KB 构建；
12. release readiness report 可被 saved-report validator 复核；
13. browser review evidence 绑定普通用户、管理员和公开入口复核；
14. 当前 release 不依赖口头确认。

二阶段知识治理验收：

1. `RetrievalQualityReport` 已进入 runtime search tools；
2. `retrieval_failures` 可记录线上和 eval 失败样本；
3. eval 输出 expected evidence hit@k；
4. eval 输出 required evidence type coverage；
5. eval 输出 source / edition diversity；
6. P0 retrieval failures 有人工处理状态；
7. 专家能复核 expected evidence、forbidden conclusion、version note、
   commentary link；
8. KB version diff 能显示知识库修改前后召回质量变化；
9. low confidence evidence 有清单；
10. Agent 生成的修正建议不能绕过人工审批；
11. 新 source、新 alias、新 term、新 commentary link 都能追溯到采纳记录；
12. release gate 必须纳入最低召回质量阈值，并在 production-ready 报告中作为
    blocker；
13. RQA schema migration 可在现有 DB 上幂等执行，失败不破坏既有数据；
14. RQA 写入、audit append、admin 状态更新和重复请求有明确失败/冲突语义；
15. backup、restore、retention、prune 和 live release 复测均覆盖 RQA 数据；
16. RQA 默认只保存脱敏摘要、hash、bounded excerpt 和枚举状态；
17. RQA admin API / Action 有 schema version、分页、最大 page size 和兼容测试；
18. metrics、Prometheus 和 release report 不输出原始 query、完整 question 或
    高基数字段列表；
19. RQA 写入、admin 查询和 release gate 有默认性能预算，缺失或超限时不能
    production-ready；
20. operator runbook、告警规则、回滚步骤和 post-release monitor 证据被 release
    report 或 validator 复核；
21. emergency disable、degraded mode、capacity smoke 和 audit history integrity
    都被 release gate 或 saved report validator 复核；
22. RQA admin read/list/update、失败访问、限流拒绝和 filter/sort allowlist 都有
    审计和 contract smoke 覆盖。

## 下一步 checklist

建议按以下顺序实现：

```text
RQA-1  Runtime search tools 输出 RetrievalQualityReport
RQA-2  Eval 增加 expected_evidence_hit@k
RQA-3  Eval 增加 required_type_coverage
RQA-4  Runtime schema 增加 retrieval_failures
RQA-5  Admin trace 暴露 retrieval quality summary
RQA-6  Agent 聚类 retrieval_failures 并生成修正建议
RQA-7  人工复核 aliases / terms / commentary_links / version_notes patch
RQA-8  KB rebuild 后按 kb_version 对比 eval 指标
RQA-9  Release gate 增加最低召回质量阈值
RQA-10 普通用户反馈进入 retrieval failure 候选，不直接修改知识库
RQA-11 管理员可以把 trace/package 标记为待专家复核
RQA-12 专家复核结论写入知识治理任务，而不是直接覆盖事实层
```

## 最终路线口径
