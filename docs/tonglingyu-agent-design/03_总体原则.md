# 03 总体原则

## 原则一：研究型优先

系统定位为研究型助手。回答必须围绕文本、脂批、版本和证据，不追求扮演人物或生成戏剧化对话。

## 原则二：证据先于表达

复杂回答必须先有证据包，再有最终回答。没有证据时，应明确说明证据不足，而不是编造。

## 原则三：正文、脂批、推断分层

不得混淆以下四类内容：

1. 正文直接写明；
2. 脂批提示；
3. 版本差异；
4. 后人推断或模型归纳。

## 原则四：Gateway 不是 Agent

Gateway 是协议适配、流程编排、安全边界和审计层，不承担文学解释和答案生成职责。它不是第 5 个 Agent。

## 原则五：Open WebUI 只暴露一个入口

用户只看到一个“通灵玉”。内部的 `honglou-main`、`honglou-text`、`honglou-commentary`、`honglou-reviewer` 不暴露给普通用户。

## 原则六：Agent 权限最小化

每个 Agent 只能调用与职责相关的工具。正文 Agent 不解释脂批，脂批 Agent 不输出最终回答，审校 Agent 不重新生成答案。

## 原则七：详细设计必须可验证

每个模块、接口、Agent 职责都必须配套验证方式。不能验证的设计暂不进入第一版。

## 原则八：错误时降级，不编造

当检索、脂批、版本比对或审校失败时，系统应降级为谨慎回答或拒绝确定结论，不得用模型记忆补齐证据。

## 原则九：业务治理层独立于 Runtime

Hermes Runtime 负责 profile、会话、工具、入口等运行能力；业务治理层负责角色权限、上下文裁剪、证据包、审计和策略。

## 原则十：第一版只做最小闭环

第一版目标是跑通：

> 问题 → 检索 → 证据包 → 分层回答 → 审校 → 返回

不在第一版扩展复杂红学论文库、全版本校勘、人物陪聊或自由创作。

## 原则十一：不要把 RAG 做成一个大向量库

正文、脂批、版本、人物事件、诗词判词、现代解释和研究观点必须分层。混在一个库里会导致系统把脂批、续书、后人观点和正文事实混为一谈。

## 原则十二：原文和规范化文本分开

原文用于引用和证据，规范化文本用于检索辅助。规范化处理不能覆盖原文，也不能丢失生僻字、异体字、旧字形和来源中已有读音。

## 原则十三：摘要不可替代证据

现代白话摘要可以帮助召回，但不能作为最终回答的可引用依据。

## 原则十四：证据必须有支持范围和不支持范围

每条证据不仅要说明它能证明什么，还要说明它不能证明什么。这个设计用于防止过度推断。

## 原则十五：先红楼专库，再古典小说通用底座

第一版先把《红楼梦》的证据分层、脂批、版本和审校跑通，再抽象到其他古典小说。
