# 18 第一版实施细化计划

## 文档定位

本文把 `16_现有架构差距与实施方向.md` 中的实施顺序细化为
可执行节点。目标不是扩大第一版范围，而是让工程实现能按顺序推进，
并在进入关键节点前暴露需要澄清的问题。

第一版仍只追求最小可靠闭环：

`source snapshot -> 知识库 -> 证据卡片 -> 证据包 -> reviewer -> 分层回答`

## 执行原则

1. 先资料和证据合同，再做 Agent 外壳；
2. 先最小可验证切片，再补广度；
3. 每个节点必须有可检查产物；
4. 每个节点必须暴露未决问题；
5. 不把风格资料、研究观点或模型解释混入事实证据层；
6. 本地生成数据默认写入 `data/tonglingyu/`，默认不提交；
7. 每次阶段完成后更新本目录 `PROGRESS.md` 或交付说明。

## 澄清项分级

| 等级 | 含义 | 处理方式 |
| --- | --- | --- |
| P0 | 不确认就不能安全进入实现 | 进入节点前必须确认 |
| P1 | 可以先做骨架，但集成前必须确认 | 以默认值推进，记录风险 |
| P2 | 不影响第一版闭环 | 放入后续规划 |

## 总体切片

推荐按以下切片推进：

1. source snapshot 下载和登记；
2. source snapshot loader；
3. SQLite/FTS 最小分层 schema；
4. 检索索引和证据卡片；
5. 证据包和核心评测；
6. Gateway、四个内部 Agent、reviewer 和审计；
7. Open WebUI 单入口；
8. 证据链稳定后再做 style profile 和多 flavor/styles。

## M0：范围锁定

目标：确保第一版实现只覆盖证据型 RAG 闭环。

产物：

1. 第一版实现范围清单；
2. 不做范围清单；
3. 当前数据目录和生成物策略；
4. 待澄清问题清单。

验收：

1. 不引用旧基础库产物；
2. 不改写 `resources/styles/` 转录和元数据；
3. 不把通用平台能力作为前置条件；
4. 不把 style profile 作为第一版上线阻塞项。

需要澄清：

| 等级 | 问题 | 默认处理 |
| --- | --- | --- |
| P0 | 第一版是否只做红楼梦，不做其他古典小说 | 默认只做红楼梦 |
| P0 | 第一版是否允许联网下载公开资料 | 默认只下载可追溯公开来源 |
| P1 | `data/tonglingyu/` 是否作为唯一生成物目录 | 默认是 |

## M1：source snapshot 下载和登记

目标：把第一批基础资料放入 `resources/sources/`，并满足
source snapshot 合同。M1 是进入 M2 的硬闸门；列入本节的内容必须完成，
否则不能进入 loader 实现。

第一批资料：

1. 维基文库《红楼梦》120 回汇校本；
2. 维基文库程甲本；
3. 维基文库程乙本；
4. 维基文库脂批整理资料；
5. 维基文库甲戌本脂评资料。

版本说明先登记在每个 snapshot 的 `metadata/source.json` 和
`docs/tonglingyu-agent-design/19_第一批资料来源登记.md`。独立影印件、
权威校注本、研究观点来源和评测题库不属于 M1 进入 M2 的前置项；它们
必须在后续证据校验或评测节点中单独列为闸门。本计划不设置未定义的
“M1.5”；任何要放在 M2 之前完成的事项，都必须明确写入 M1 闸门。

工程任务：

1. 使用 `scripts/download_wikisource.py` 下载正文；
2. 对 EPUB 或其他格式使用 `scripts/extract_epub.py`；
3. 为每个来源生成 `metadata/source.json`、`documents/blocks.jsonl` 和 hash；
4. 检查 `source_category`、`work`、`edition`、`language`；
5. 抽样检查原始字形和 `rare_char_annotations`；
6. 生成 source snapshot 入库报告；
7. 生成并通过 M1 source snapshot registry 校验。

验收：

1. 每个来源能回溯到 URL、文件 hash 或等价来源说明；
2. 每个 block 有稳定 ID 和来源位置；
3. 原始文本没有被规范化文本覆盖；
4. 来源已有读音进入 `rare_char_annotations`；
5. 风格资料不进入基础证据来源；
6. `source snapshot ready` 只代表工程上可进入 loader，不代表完成学术校勘；
7. 进入 M2 前必须通过 source snapshot registry 和抽样点校验；
8. 本批 snapshot 不保存空的 HTML 派生产物，源码级版本依据为 `.wiki`、
   `revision_id`、`source_url` 和 metadata；
9. 本节所有 P0 问题必须关闭，不能遗留到 M2。

需要澄清：

| 等级 | 问题 | 默认处理 |
| --- | --- | --- |
| P0 | 脂批本第一批具体采用哪个公开来源 | 已采用 Wikisource 脂批整理资料和甲戌本 |
| P0 | 版本说明资料的权威来源或许可边界 | 已登记 Wikisource 页面说明和版本边界 |
| P1 | 是否需要保存原始下载响应 | 默认保存来源 hash 和抽取产物 |
| P1 | 是否需要影印件作为关键引文复核材料 | 后续证据校验闸门处理，不阻塞 M2 |

## M2：source snapshot loader

目标：在 M1 闸门通过后，实现从 snapshot 到本地 SQLite 的确定性导入。

工程任务：

1. 定义 loader CLI；
2. 校验 snapshot 必填字段；
3. 校验 `documents/blocks.jsonl` 格式；
4. 校验 block ID、章节、段落类型和来源位置；
5. 导入 `sources`、`editions`、`chapters`、`blocks`；
6. 导入 `rare_char_annotations`；
7. 输出导入报告和错误报告。

验收：

1. 重复导入同一 snapshot 不产生重复事实；
2. 缺必填字段时给出结构化错误；
3. 字形读音能从 snapshot 传递到数据库；
4. loader 不生成文学解释；
5. loader 不调用模型。

需要澄清：

| 等级 | 问题 | 默认处理 |
| --- | --- | --- |
| P0 | loader 是否必须支持增量导入 | 第一版可全量重建 |
| P1 | block ID 是否由 snapshot 固定还是 loader 生成 | 默认 snapshot 固定 |
| P1 | 是否需要导入报告落盘 | 默认落盘到 `data/tonglingyu/reports/` |

## M3：SQLite/FTS 最小分层 schema

目标：建立第一版可检索、可审校、可回放的最小 schema。

最小表：

1. `sources`
2. `editions`
3. `chapters`
4. `blocks`
5. `blocks_fts`
6. `rare_char_annotations`
7. `commentaries`
8. `versions`
9. `people`
10. `aliases`
11. `relationships`
12. `events`
13. `poems`
14. `terms`
15. `evidence_cards`
16. `evaluation_cases`
17. `kb_version`

工程任务：

1. 定义 schema migration；
2. 建立 FTS 表和同步策略；
3. 建立必要唯一约束；
4. 建立版本和来源外键；
5. 建立 seed 数据格式；
6. 增加 schema smoke test。

验收：

1. 能从空库初始化；
2. 能导入正文 snapshot；
3. 能按回目、人物名、诗句检索；
4. 能查询来源和版本边界；
5. 能保留原文和规范化检索文本的分离。

需要澄清：

| 等级 | 问题 | 默认处理 |
| --- | --- | --- |
| P0 | 第一版是否只使用 SQLite，不接外部向量库 | 默认只用 SQLite/FTS |
| P1 | schema migration 工具是否自研 | 默认用项目内轻量 SQL |
| P1 | 人物和事件 seed 首批规模 | 默认人物 30 到 50，事件 50 到 80 |

## M4：检索索引和证据卡片

目标：让 RAG 层返回结构化证据卡片，而不是最终答案。

工程任务：

1. 实现正文精确检索；
2. 实现 FTS 检索；
3. 实现别名索引；
4. 实现脂批和版本检索接口；
5. 实现反证和限制索引；
6. 定义 evidence card schema；
7. 强制 `support_scope` 和 `unsupported_scope`；
8. 标注 `verification_status` 和 `confidence`。

验收：

1. 原文定位问题能返回回目、段落和原文片段；
2. 脂批问题能返回批语版本和对应正文；
3. 人物别名能回到标准人物；
4. 证据卡片能引用 `rare_char_annotations`；
5. 现代白话摘要不能作为可引用证据；
6. 每张证据卡片都有不支持范围。

需要澄清：

| 等级 | 问题 | 默认处理 |
| --- | --- | --- |
| P0 | 证据卡片字段名是否需要对外稳定 | 默认第一版内部稳定 |
| P1 | 是否需要保留证据卡片落库 | 默认落库便于回放 |
| P1 | `confidence` 是否先用规则枚举 | 默认高、中、低 |

## M5：证据包和核心评测

目标：让复杂回答可以被 reviewer 审校和回放。

工程任务：

1. 定义 evidence package schema；
2. 定义 claim statement schema；
3. 建立 20 到 30 条首批核心评测；
4. 覆盖正文、脂批、版本、人物别名、诗词判词；
5. 覆盖字形读音和证据不足；
6. 输出评测运行报告；
7. 逐步扩展到 100 到 200 条评测。

验收：

1. 每个复杂问题有证据包编号；
2. 每个结论声明能回溯到证据卡片；
3. 评测题包含预期证据和禁止结论；
4. 证据不足问题不会被强答；
5. 评测报告能比较不同 `kb_version`。

需要澄清：

| 等级 | 问题 | 默认处理 |
| --- | --- | --- |
| P0 | 首批评测题是否由领域专家确认 | 先工程草案，集成前确认 |
| P1 | 评测输出格式是否需要兼容仪表盘 | 第一版先 jsonl |
| P1 | 是否需要人工标注 expected evidence ID | 默认需要 |

## M6：Gateway、四个 Agent、reviewer 和审计

目标：实现固定编排闭环，且用户不能跳过审校。

工程任务：

1. 实现 OpenAI-compatible Gateway 入口；
2. 实现会话映射和追踪编号；
3. 定义 `honglou-main` profile；
4. 定义 `honglou-text` profile；
5. 定义 `honglou-commentary` profile；
6. 定义 `honglou-reviewer` profile；
7. 实现证据包输入输出约束；
8. 实现 reviewer 拦截和修订循环；
9. 实现审计日志。

验收：

1. Open WebUI 只能看到一个通灵玉入口；
2. 用户不能指定内部 Agent；
3. 用户不能关闭 reviewer；
4. 每次请求有 trace ID；
5. 复杂回答有 evidence package ID；
6. reviewer 能拦截无证据断言、版本混淆和过度推断；
7. 下游失败时能降级为谨慎回答或证据不足。

需要澄清：

| 等级 | 问题 | 默认处理 |
| --- | --- | --- |
| P0 | 第一版模型后端使用哪个现有 Hermes/OpenAI 配置 | 进入实现前确认 |
| P0 | Gateway 部署在本地还是沿用现有 deploy 栈 | 进入实现前确认 |
| P1 | 审计日志是否落 SQLite 还是文件 | 默认 SQLite 加文件报告 |
| P1 | reviewer 拦截后是否允许一次自动修订 | 默认允许一次 |

## M7：Open WebUI 单入口

目标：让用户只通过“通灵玉”模型使用第一版能力。

工程任务：

1. 新增 Open WebUI 可见模型配置；
2. 隐藏或限制内部模型入口；
3. 配置 Gateway endpoint；
4. 验证普通用户权限；
5. 验证管理员可追踪证据包；
6. 验证 Cloudflare Tunnel 只通向 Open WebUI。

验收：

1. 普通用户只看到“通灵玉”；
2. Open WebUI 不暴露内部 Agent；
3. 公开入口不暴露 Hermes `8642` 或 `9119`；
4. 管理员能按 trace ID 找到证据包和审校记录；
5. 失败时不泄露内部异常栈。

需要澄清：

| 等级 | 问题 | 默认处理 |
| --- | --- | --- |
| P0 | 是否直接修改当前远端 Open WebUI 配置 | 实施前确认 |
| P0 | 是否需要先做本地 dry run | 默认先本地或隔离环境验证 |
| P1 | 模型显示名是否固定为“通灵玉” | 默认是 |

## M8：style profile 和多 flavor/styles

目标：在证据链稳定后，再把风格能力结构化。

工程任务：

1. 定义 `style_profile.json` 合同；
2. 定义 `style_rules.json` 合同；
3. 为 `buhongjushi` 生成只读 style profile；
4. 增加默认 `neutral_research` profile；
5. 增加 style eval cases；
6. 确保风格约束进入草稿生成前；
7. 确保 reviewer 检查风格化后的表述。

验收：

1. 风格资料不进入正文、脂批、版本或校订证据层；
2. 风格不会改写原文引文；
3. 风格不会删除必要的版本和证据不足提示；
4. 风格不会引入新事实；
5. 不复制长段转录文本；
6. 不在 reviewer 后单独改写最终答案。

需要澄清：

| 等级 | 问题 | 默认处理 |
| --- | --- | --- |
| P1 | 第一版是否开放用户选择风格 | 默认不开放 |
| P1 | `buhongjushi` 是否只抽象风格，不复用观点 | 默认只抽象风格 |
| P2 | 后续新增哪些 flavor/styles | 放入 V2 后评估 |

## 阶段交付节奏

建议每完成一个节点输出一次小结：

1. 已完成产物；
2. 运行过的检查；
3. 新发现的风险；
4. 需要用户确认的问题；
5. 下一节点是否可以开始。

如果出现 P0 澄清项，应停止进入对应实现节点，但可以继续做不依赖该
决定的文档、测试夹具或接口骨架。

## 当前最早需要确认的问题

截至第一批资料下载，以下问题已有默认决策：

1. 第一批资料从维基文库获取；
2. 允许联网下载公开资料；
3. 第一版只使用 SQLite/FTS，不接外部向量库；
4. 远程部署复用一个现有 Open WebUI；
5. Gateway 单独部署。

仍需在后续节点前确认：

1. 是否引入影印件作为关键引文复核材料；
2. 脂批资料是否需要人工拆分批语类型；
3. Gateway 远程部署复用哪个现有网络和数据库边界。

第一批资料来源登记见 `19_第一批资料来源登记.md`。
