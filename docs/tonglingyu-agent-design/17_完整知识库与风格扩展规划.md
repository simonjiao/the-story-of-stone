# 17 完整知识库与风格扩展规划

## 文档定位

本文用于规划“通灵玉”第一版稳定后的知识库和风格能力演进。
它不是第一版上线前置条件，也不替代 `08_知识库与RAG设计.md`、
`12_验证方案与验收标准.md` 和 `14_迭代路线图.md`。

第一版仍以最小可靠闭环为主：

`source snapshot -> 知识库 -> 证据卡片 -> 证据包 -> reviewer 审校 -> 分层回答`

本文讨论的是在这条闭环稳定后，如何扩展到更完整的红楼梦专库、更多表达风格和后续古典小说底座。

## 基本判断

完整知识库不是一个更大的向量库，而是一个更完整的证据治理系统。它应当继续遵守以下原则：

1. source snapshot 是所有入库资料的入口；
2. 原文、脂批、版本、人物关系、事件、诗词判词、研究观点和风格资料必须分层；
3. 原始字形和来源中已有读音必须保留；
4. 规范化文本、摘要和 embedding 只用于检索辅助，不能替代可引用证据；
5. 模型生成内容不得自动反写为知识库事实；
6. 风格资料只影响表达方式和讲解路径，不进入正文、脂批、版本或校订证据层；
7. reviewer 必须能检查证据是否支持最终表述，包括风格化表达后的表述。

## 阶段规划

### V1：红楼梦证据专库稳定

目标是完成一个可上线的红楼梦证据型知识库。

范围：

1. 120 回通行本文本，明确前八十回和后四十回边界；
2. 关键脂批，优先覆盖人物命运、伏笔和版本相关批语；
3. 通行本、脂评系统、程甲和程高系统的基础说明；
4. 第五回判词、曲文、册页图画和关键诗词；
5. 30 到 50 个核心人物；
6. 50 到 80 个核心事件；
7. 200 条左右标准问题和负面用例；
8. `rare_char_annotations` 从 source snapshot 传递到证据卡片。

验收重点：

1. 正文定位稳定；
2. 脂批来源和对应正文可追溯；
3. 版本边界清楚；
4. 人物别名和关系可回溯到证据；
5. 诗词判词能独立命中；
6. 证据不足时不强答；
7. reviewer 能拦截过度推断。

### V1.5：知识库治理增强

目标是在不改变第一版产品边界的前提下，提高维护、排错和验收能力。

候选能力：

1. `kb_version` 版本记录和可回滚构建产物；
2. 证据卡片 diff 和证据包回放；
3. source snapshot 入库报告；
4. 未校验证据、低置信证据和来源缺失证据清单；
5. 评测题库分组、标签和回归报告；
6. 管理员证据包查询界面；
7. 检索失败样本收集；
8. reviewer 拦截原因统计。

本阶段仍不引入自由学习入库，也不把用户聊天内容直接沉淀为知识库事实。

### V2：扩展资料层和研究观点层

目标是从“能可靠回答核心问题”扩展到“能解释资料来源和观点差异”。

候选能力：

1. 增加更多可追溯版本来源；
2. 增加红学研究观点库；
3. 增加研究观点与正文、脂批、版本证据之间的关联；
4. 增加引用导出；
5. 增加专题问答；
6. 增加教学模式；
7. 增加证据链展开界面；
8. 增加评测仪表盘。

研究观点库必须单独分层。它可以解释“某种观点认为”，但不能覆盖正文、脂批或版本事实。回答中引用研究观点时，必须区分：

1. 正文直接证据；
2. 脂批提示；
3. 版本差异；
4. 研究者观点；
5. 系统合理推断；
6. 证据不足。

### V3：校勘、标注和知识生产工具

目标是支持专家参与和更复杂的资料治理。

候选能力：

1. 多用户协作标注；
2. 人工校勘工作台；
3. 版本差异可视化；
4. 红学观点谱系；
5. 证据链人工修订；
6. 教学课程生成；
7. 文本细读辅助；
8. 研究报告生成。

本阶段可以产生新标注，但标注进入知识库前必须有状态流转：

`草稿 -> 待审 -> 已审 -> 可引用`

未审核标注只能作为辅助材料，不能作为强证据。

### V4：古典小说通用底座

目标是在红楼梦专库稳定后，抽象古典小说通用底座。

扩展顺序建议：

| 阶段 | 作品或能力 | 专属知识层 |
| --- | --- | --- |
| 第一阶段 | 红楼梦 | 脂批库、判词曲文库、版本库 |
| 第二阶段 | 三国演义 | 历史对照库、战役事件库、势力变迁库 |
| 第三阶段 | 水浒传 | 绰号库、座次库、上山事件库 |
| 第四阶段 | 西游记 | 神魔实体库、路线库、法宝库 |
| 第五阶段 | 跨书比较 | 母题库、人物类型库、叙事结构库 |

跨书比较必须建立在各作品专库成熟之后，不应在早期用一个混合大库直接做多书 RAG。

## 完整知识库对象

完整知识库建议分为以下对象群。

### 来源与版本对象

| 对象 | 职责 |
| --- | --- |
| `sources` | 登记 source snapshot、来源类型、hash、URL、抽取时间 |
| `source_documents` | 登记 snapshot 中的文档和 block 文件 |
| `editions` | 登记版本、版本系统、存回范围和使用限制 |
| `kb_version` | 登记一次知识库构建的输入、输出、schema 和校验结果 |

### 原文与文本对象

| 对象 | 职责 |
| --- | --- |
| `chapters` | 回次、回目、版本范围 |
| `blocks` | 原文段落、位置、段落类型、原始字形 |
| `blocks_fts` | 检索规范化文本 |
| `terms` | 关键词、专名、物件、地点 |
| `rare_char_annotations` | 生僻字、异体字、旧字形和来源读音 |

### 脂批与版本对象

| 对象 | 职责 |
| --- | --- |
| `commentaries` | 批语文本、批语类型、批者标签、来源版本 |
| `commentary_links` | 批语与正文 block 的对应关系 |
| `version_notes` | 版本说明、来源状态、使用限制 |
| `version_differences` | 版本差异、差异范围和证据等级 |

### 人物、关系和事件对象

| 对象 | 职责 |
| --- | --- |
| `people` | 标准人物、身份、首次出现、说明 |
| `aliases` | 别名、称谓、使用范围 |
| `relationships` | 人物关系、关系等级、版本范围、支撑证据 |
| `events` | 事件名称、回次、参与人物、前因后果、主题标签 |
| `event_links` | 事件与正文、人物、诗词、脂批的关联 |

### 专题文本对象

| 对象 | 职责 |
| --- | --- |
| `poems` | 诗、词、曲、判词、灯谜、偈语 |
| `poem_interpretations` | 受控解释、关联人物、证据等级 |
| `topic_collections` | 第五回判词、葬花词、太虚幻境等专题集合 |

### 研究观点对象

| 对象 | 职责 |
| --- | --- |
| `research_sources` | 研究资料来源、作者、出版信息和使用限制 |
| `research_claims` | 研究观点、适用范围、反对意见、证据关联 |
| `claim_evidence_links` | 研究观点与正文、脂批、版本证据的链接 |

研究观点对象默认不进入第一版核心能力。

### 证据治理对象

| 对象 | 职责 |
| --- | --- |
| `evidence_cards` | 统一证据卡片，包含支持范围和不支持范围 |
| `evidence_packages` | 一次回答使用的证据集合 |
| `claim_statements` | 结论声明、证据等级、限制条件 |
| `review_records` | reviewer 审校结果、问题列表和处理状态 |

### 评测对象

| 对象 | 职责 |
| --- | --- |
| `evaluation_cases` | 标准问题、预期证据、禁止结论、审校预期 |
| `evaluation_runs` | 一次评测运行的模型、知识库版本和结果 |
| `retrieval_failures` | 误召回、漏召回和证据不足误判样本 |

## 检索演进

检索能力建议按以下顺序增强：

1. 精确索引：人名、回目、诗句、判词、专名；
2. FTS：正文和批语的规范化检索；
3. 别名索引：凤姐、宝二爷、林姑娘等称谓；
4. 结构索引：回目、事件、人物线索、地点和主题；
5. 脂批和版本索引：批语、对应正文、版本范围；
6. 反证和限制索引：支撑“不能证明”和“证据不足”；
7. 语义召回：解释型、主题型、关系型问题；
8. 重排和证据包构造：按相关性、证据等级、来源可靠性排序。

embedding 可以作为召回手段，但它返回的内容必须被还原为具体证据卡片。不能把 embedding 命中的摘要直接作为回答证据。

## 风格和 Flavor 规划

### 定位

风格和 flavor 是表达层能力，不是事实层能力。它们用于决定回答的讲解方式、节奏、结构和措辞，不用于决定证据真伪。

运行时应保持以下顺序：

`证据检索 -> 证据包 -> 带风格约束的草稿 -> reviewer 审校 -> 最终回答`

风格不能在 reviewer 之后单独改写最终回答。否则风格层可能引入未经审校的新断言。

### 目录建议

后续可以继续沿用 `resources/styles/` 边界：

```text
resources/styles/
  buhongjushi/
  academic_plain/
  beginner_friendly/
  teacher_mode/
  concise_research/
  classical_commentary/
```

每个风格目录建议包含：

| 文件或对象 | 用途 |
| --- | --- |
| `README.md` | 风格来源、适用范围、禁止范围 |
| `metadata/manifest.json` | 来源清单、hash、抽取时间 |
| `transcripts/` | 原始转录或示例资料 |
| `style_profile.json` | 结构化风格摘要 |
| `style_rules.json` | 可执行风格约束 |
| `style_examples.jsonl` | 可引用的短示例和反例 |
| `style_eval_cases.jsonl` | 风格回归测试 |

### Style Profile 合同

每个 style profile 至少包含：

1. `style_id`
2. `display_name`
3. `source_category = style_material`
4. 来源说明和使用许可说明；
5. 适用问题类型；
6. 语气约束；
7. 结构约束；
8. 引用和证据展示偏好；
9. 禁止行为；
10. 风格评测用例。

禁止行为至少包括：

1. 不新增事实；
2. 不改写原文引文；
3. 不替代证据层；
4. 不把风格资料中的观点当作正文事实；
5. 不复制长段原始表达；
6. 不绕过 reviewer；
7. 不削弱版本、脂批和证据不足提示。

### 风格候选

| 风格 | 用途 | 风险 |
| --- | --- | --- |
| `neutral_research` | 默认研究型回答 | 可能偏硬，需要保证普通用户可读 |
| `beginner_friendly` | 面向初读者 | 不能过度简化证据边界 |
| `academic_plain` | 面向研究型用户 | 不能堆砌术语 |
| `teacher_mode` | 面向教学讲解 | 不能变成课程生成器后忽略证据 |
| `concise_research` | 短答和快速定位 | 不能省略关键版本限制 |
| `buhongjushi` | 借鉴现有风格资料的讲解节奏 | 只能抽象表达方式，不能替代证据或复制转录文本 |
| `classical_commentary` | 偏古典评点式表达 | 不能伪造批语或仿作原文 |

第一版可以只保留默认研究型表达和现有 `buhongjushi` 资料边界，不必开放用户自由选择风格。多风格切换适合在证据链和 reviewer 稳定后再引入。

## 风格与知识库的关系

风格资料可以进入 source snapshot 分类中的 `style_material`，但它不进入以下证据层：

1. 正文证据；
2. 脂批证据；
3. 版本证据；
4. 校勘证据；
5. 人物关系事实；
6. 事件事实。

风格资料可以影响：

1. 回答结构；
2. 解释顺序；
3. 比喻和过渡；
4. 面向用户的术语解释；
5. 证据展示密度；
6. 是否先给结论或先铺证据。

## 验收标准

完整知识库扩展的验收标准：

1. 每个新增来源都能回溯到 source snapshot；
2. 每个新增证据类型都有支持范围和不支持范围；
3. 新增研究观点不会覆盖正文、脂批和版本证据；
4. 新增检索能力不会提高混库污染率；
5. 新增风格不会改写原文引文；
6. 新增风格不会删除必要的版本和证据不足提示；
7. reviewer 能检查风格化回答中的过度断言；
8. 评测集能比较不同 `kb_version` 和 `style_id` 的表现。

## 不做范围

未来规划仍然不建议做以下事项：

1. 把所有网络资料直接入库；
2. 让用户问题自动变成知识库事实；
3. 用模型记忆补齐缺失版本信息；
4. 把现代白话摘要作为可引用证据；
5. 把风格资料当作红学研究观点；
6. 把红学研究观点当作正文事实；
7. 在红楼梦专库稳定前做多书混合 RAG；
8. 在 reviewer 稳定前开放用户自定义内部 Agent 或任意风格 prompt。

## 近期落地建议

近期仍应按以下顺序推进：

1. 完成第一版 source snapshot 下载和登记；
2. 完成最小 SQLite/FTS 分层 schema；
3. 完成证据卡片和证据包；
4. 完成核心评测题；
5. 完成 Gateway、四个内部 Agent、reviewer 和审计；
6. 等证据链稳定后，再实现 style profile 结构化配置；
7. 最后开放多 flavor/styles 的受控选择。

判断标准是：先让系统答得有证据，再让系统答得更好懂、更有风格。
