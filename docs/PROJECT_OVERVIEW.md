# 项目概览

本项目的长期目标是构建一个可修订、可追溯、可自定义风格的《红楼梦》交互机器人。当前阶段先整理《红楼梦》相关视频转录和校订流程，支持后续多 agent 协作校订。

当前第一版产品方向命名为“通灵玉”：一个面向《红楼梦》的研究型 Hermes Agent，通过 Open WebUI 暴露单一入口，内部由 Gateway 编排 `honglou-main`、`honglou-text`、`honglou-commentary`、`honglou-reviewer` 四个受控 Agent，输出区分正文、脂批、版本差异和推断层级的回答。

## 项目做什么

- 下载 B 站“红楼梦文本探究”合集视频。
- 从视频提取音频，并生成机器转录文本、字幕和转录 JSON。
- 对转录文本中的原文引用、脂批、章回名、术语和同音词做交叉验证。
- 将新 EPUB、维基文库《红楼梦》全本、脂批本和版本说明资料抽取为可登记的 source snapshot，作为后续知识库构造输入。
- 规划一个跨机器可访问的校验服务，给多个 agent 共用同一套已登记文本证据、研究资料、风格语料和校订记录。
- 为后续交互机器人准备可选择的对话风格和讨论维度。
- 维护“通灵玉”第一版产品、架构、接口、权限审计和验收设计文档。

## 资料分类

- 基础资料：后续重新选定并登记的《红楼梦》文本证据，可直接引用和用于校订。
- 扩展基础资料：后续新增的其他《红楼梦》发行版本，同样可直接引用，用于版本比较。
- 研究资料：后续新增的论文、专著摘录、讲义、札记等，用于观点和研究脉络。
- 风格资料：视频转录、讲稿、解读文章等，只影响表达方式、讲解习惯和分析路径。
- 校订记录：用户或 agent 确认过的修改、证据和判断。

## 主要输入

- `https://space.bilibili.com/558777092/lists`: B 站空间合集来源。
- `https://zh.wikisource.org/`: 《红楼梦》全本、脂批本等基础资料来源。
- `resources/hongloumeng_asr_glossary.txt`: ASR 术语词表。
- 新选定的 EPUB、脂批、版本说明和人物事件资料。

## 主要输出

- `resources/styles/buhongjushi/transcripts/`: 每条视频的 `.txt`、`.srt`、`.transcript.json`。
- `resources/styles/buhongjushi/metadata/`: 视频合集清单和运行状态。
- `resources/sources/`: 新资料抽取后的 source snapshot。

## 风格资料

B 站“红楼梦文本探究”视频转录是当前第一批讲解风格语料，项目内命名为 `不红居士` 风格。

这个风格名用于后续交互机器人的表达方式和讲解习惯配置，不替代转录原文中的讲解者自称。转录文本里的 `不红君` 仍按原内容保留。风格资料附加在基础资料之上，不能覆盖原文、注释、批语、图片字形或校订记录。

## 通灵玉设计文档

`docs/tonglingyu-agent-design/` 保存红楼研究型 Hermes Agent 第一版的渐进式设计文档。阅读入口是 `00_阅读路径与文档地图.md`，设计范围覆盖产品定位、用户流程、证据分层、四 Agent 分工、Gateway、知识库与 RAG、接口契约、权限审计、验证方案、负面清单、迭代路线和命名语义。

## 核心原则

- 转录文本是机器初稿，不能直接当作可引用文本。
- 原文引用、脂批和章回名必须回到明确登记的文本证据确认。
- 一个视频可能跨多个章节，校验时不能只查视频标题对应章节。
- 向量召回只负责快速找到可能相关段落，最终校订必须落到可追溯证据。
- 新资料先进入 source snapshot，再由知识库构造流程登记为正文、脂批、版本或人物事件证据。
- 风格语料只能影响表达方式，不能覆盖原文证据和校订规则。
